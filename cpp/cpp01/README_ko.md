## 🛠️ General Rules
과제를 수행하며 반드시 준수해야 하는 엄격한 규칙들입니다. 이 규칙을 어기면 실력이 늘지 않을 뿐더러 점수도 처참해질 거야!

* **메모리 관리**: `new`를 사용해 할당한 메모리는 반드시 `delete`로 해제하여 **Memory Leak**을 방지해야 합니다.

### 💡 Deep Dive

#### **Q1. 힙(Heap) 할당은 언제 사용하나요?**
* **답변**: 객체가 함수가 종료된 후에도 살아있어야 하거나, 컴파일 타임에 크기를 알 수 없는 큰 데이터를 다룰 때 사용합니다. `new` 키워드를 사용하며, 반드시 `delete`로 직접 수동 해제해주어야 합니다.

#### **Q2. 스택(Stack) 할당의 장점은 무엇인가요?**
* **답변**: 관리가 매우 쉽고 빠릅니다. 함수의 중괄호(`{ }`)를 벗어나는 순간 시스템이 자동으로 메모리를 회수하고 소멸자를 호출해주기 때문입니다. 메모리 누수 위험이 없어 가능하면 스택 할당을 우선적으로 고려하는 것이 좋습니다.

#### **Q3. 소멸자에서 이름을 출력하게 하는 이유는 무엇인가요?**
* **답변**: 프로그래머가 메모리 해제를 적절한 타이밍에 했는지 시각적으로 확인하기 위해서입니다. 힙에 할당된 좀비를 `delete` 하지 않으면 프로그램 종료 시까지 메시지가 출력되지 않으며, 이는 곧 메모리 누수를 의미합니다.

#### **Q4. `Zombie z(name);`와 `Zombie z = Zombie(name);`은 똑같은 건가요?**
* **답변**: 결과는 같아 보이지만 미묘한 차이가 있습니다. 첫 번째는 **직접 초기화(Direct Initialization)**로, 즉시 객체를 생성합니다. 두 번째는 **복사 초기화(Copy Initialization)**로, 임시 객체를 만든 뒤 이를 복사하는 과정을 거칠 수 있습니다. C++98 표준에서는 성능과 명확성을 위해 첫 번째 방식을 더 권장합니다.

#### **Q5. C언어의 `malloc`이 있는데 왜 C++에서는 `new`를 쓰나요?**
* **답변**: `malloc`은 단순히 메모리 덩어리만 할당할 뿐이지만, `new`는 연산자로서 **객체의 생성자를 자동으로 호출**해줍니다. 즉, 메모리 할당과 동시에 객체를 사용할 준비(초기화)까지 마쳐주기 때문에 C++의 객체 지향 철학에 필수적입니다. 반대로 `delete`는 메모리 해제 전 **소멸자를 호출**해 뒷정리를 도와줍니다.

#### **Q6. 함수가 끝났는데 `delete`를 안 하면 어떻게 되나요?**
* **답변**: 힙(Heap)에 할당된 메모리는 프로그래머가 직접 관리하기로 약속한 영역입니다. 따라서 `delete`를 호출하지 않으면 프로그램이 끝나도 해당 객체의 **소멸자가 실행되지 않으며**, 이는 메모리 누수(Memory Leak)로 이어집니다. "Destroyed" 메시지가 출력되지 않는다면 바로 이 상황입니다.

#### **Q7. 왜 함수 이름이 `randomChump`인가요?**
* **답변**: "Chump"는 엑스트라나 만만한 사람을 뜻하는 속어입니다. 이 함수는 좀비를 **스택(Stack)**에 생성하여 함수가 끝나면 바로 사라지게 만듭니다. 즉, 주인공처럼 계속 살아남는 좀비가 아니라 잠깐 나타났다 사라지는 '별 볼 일 없는 좀비'라는 의미를 할당 방식의 특성에 빗대어 위트 있게 표현한 것입니다.

#### **Q8. 생성자 본문(`{}`) 대신 `: name(name)`처럼 초기화 리스트를 쓰는 이유는 무엇인가요?**
* **답변**: C++에서 객체가 생성되는 과정은 '초기화'와 '대입'으로 나뉩니다.
* **효율성 (Initialization vs Assignment)**: 초기화 리스트(`: name(name)`)를 사용하면 객체가 메모리에 올라감과 동시에 값이 설정됩니다. 반면 `{ name = name; }`처럼 본문에서 대입하면, 일단 빈 값으로 객체가 생성된 후 다시 값을 덮어씌우는 과정을 거치므로 비효율적입니다.
* **필수적인 상황**: 다음과 같은 경우 초기화 리스트 사용이 선택이 아닌 **필수**입니다.
	1.  **`const` 멤버 변수**: 상수는 생성과 동시에 값이 정해져야 하며, 생성자 본문에서의 대입(수정)이 불가능합니다.
	2.  **참조자(`&`) 멤버 변수**: 참조자는 선언과 동시에 대상을 가리켜야 하므로 초기화 리스트에서 정의해야 합니다.
	3.  **기본 생성자가 없는 클래스 멤버**: 멤버 변수로 포함된 다른 클래스가 인자 없는 생성자를 지원하지 않을 때, 어떻게 초기화할지 리스트에 명시해야 합니다.
* **가독성**: 객체의 기반이 되는 데이터 설정을 로직 실행부(`{}`)와 분리하여 코드의 의도를 명확히 전달합니다.
```cpp
/* Zombie.cpp */

Zombie::Zombie(std::string name) : name(name) // 초기화 리스트: 탄생과 동시에 이름 부여
{
	// 생성자 본문: 초기화 이후 실행될 로직이 들어가는 곳
}
```

#### **Q9. `Zombie::Zombie(std::string name) : name(name)`에서 왜 같은 이름을 두 번 쓰나요?**

* **답변**: 이것은 C++에서 **멤버 변수**와 **매개변수**의 이름이 같을 때 발생하는 '이름 가리기(Name Shadowing)' 문제를 가장 깔끔하게 해결하는 방법입니다.
* **역할 분담**:
	* **앞의 `name` (콜론 뒤)**: 클래스 내부(`private`)에 선언된 **멤버 변수**를 가리킵니다. 컴파일러는 이 위치에 오는 이름을 무조건 멤버 변수로 인식하도록 약속되어 있습니다.
	* **뒤의 `(name)` (괄호 안)**: 생성자의 인자로 전달된 **매개변수**를 가리킵니다.
* **장점**:
	* `this->name = name;`과 같은 추가적인 대입 연산 없이, 객체가 생성되는 시점에 인자 값을 멤버 변수에 직접 주입합니다.
	* `newName`이나 `pName` 처럼 억지로 이름을 바꿀 필요가 없어, 변수의 의미(이름을 설정한다)를 더 명확하게 유지할 수 있습니다.

```cpp
/* Zombie.cpp */

// (std::string name) <- 외부에서 전달된 '매개변수'
Zombie::Zombie(std::string name)
	: name(name) // 앞의 name: 클래스의 멤버 변수 / 뒤의 (name): 받은 매개변수
{
	// 초기화 리스트를 통해 '내 주머니(멤버 변수)'에 '네가 준 것(매개변수)'을
	// 탄생과 동시에 넣었으므로, 이 중괄호 안은 비워두어도 됩니다.
}
```

#### **Q10. 멤버 변수 이름 앞에 언더바(`_`)를 붙이는 특별한 이유가 있나요?**

* **답변**: 네, 이것은 **명명 규칙(Naming Convention)**의 하나로, 코드의 가독성을 높이고 실수를 방지하기 위해 사용합니다.
* **주요 이점**:
	1.  **시각적 구분**: 코드 어디서든 `_name`을 보는 순간, "이것은 함수 내부의 지역 변수가 아니라 클래스의 멤버 변수구나"라고 즉시 인지할 수 있습니다.
	2.  **`this->` 포인터 생략 가능**: 멤버 변수와 매개변수의 이름이 다르면(`_name` vs `name`), `this->name = name;`처럼 길게 쓰지 않고 `_name = name;`만으로도 혼동 없이 대입이 가능합니다.
	3.  **초기화 리스트의 명확성**: `_name(name)`과 같이 작성하면, 어느 쪽이 클래스의 소속이고 어느 쪽이 외부에서 받은 값인지 훨씬 직관적으로 보입니다.
* **⚠️ 주의 사항 (시스템 예약어)**:
	* C++ 표준에서 **언더바 + 대문자(`_Name`)** 또는 **이중 언더바(`__name`)**로 시작하는 식별자는 컴파일러나 표준 라이브러리가 사용하기 위해 예약해 둔 영역입니다.
	* 따라서 충돌을 피하기 위해 반드시 **언더바 + 소문자(`_name`)** 형식을 사용하거나, 변수 뒤에 언더바를 붙이는 방식(`name_`)을 사용하는 것이 안전합니다.

```cpp
/* Zombie.hpp */

class Zombie {
private:
	std::string _name; // 멤버 변수는 _로 시작 (가독성 UP!)

public:
	Zombie(std::string name) : _name(name) {} // 초기화 리스트가 훨씬 명확해짐
};
```


### 🧠 메모리 구조 다시보기 (Memory Segments)

객체가 저장되는 위치를 이해하는 것은 C++ 설계의 시작입니다.

* **Stack**: 지역 변수, 매개변수. `{}`를 벗어나면 자동 소멸. (`randomChump`)
* **Heap**: `new`로 할당한 동적 메모리. `delete` 전까지 유지. (`newZombie`)
* **Data/BSS**: 전역 변수, 정적(static) 변수가 저장되는 공간. 프로그램 종료 시까지 유지.

### 💡 Deep Dive

#### **Q1. Getter와 Setter는 왜 사용하며, ex01에서는 왜 setName이 필요한가요?**
* **답변**: 객체의 내부 데이터(`_name`)를 외부에서 직접 수정하지 못하게 보호(캡슐화)하기 위해 사용합니다.  특히 `new[]` 연산자는 인자 없는 **기본 생성자**만 호출할 수 있기 때문에, 객체 생성 직후 이름을 부여하기 위해서는 **Setter(`setName`)** 함수가 반드시 필요합니다.

#### **Q2. Getter와 Setter 중 무엇을 더 지양해야 하며, 그 이유는 무엇인가요?**
* **답변**: **세터(Setter)**의 사용을 더 지양해야 합니다. 세터를 남발하면 외부에서 객체의 상태를 마음대로 바꿀 수 있어 캡슐화가 깨지고, 객체의 자율성이 훼손되기 때문입니다. 가급적 생성자를 통해 초기 상태를 확정하는 것이 좋습니다.

#### **Q3. Getter와 Setter를 반드시 쌍으로 만들어야 하나요?**
* **답변**: 아닙니다. '읽기 전용' 속성에는 게터만, 외부 노출이 필요 없는 '설정' 속성에는 세터만 만드는 등 설계 의도에 따라 독립적으로 사용합니다. 단, 게터 선언 시에는 객체를 수정하지 않는다는 의미로 함수 뒤에 **`const`**를 붙이는 습관(Const Correctness)이 중요합니다.

#### **Q4. Getter/Setter와 얕은/깊은 복사는 어떤 차이가 있나요?**

* **답변**: 현재 구현(`zombieHorde`)에서는 사용되지 않지만, 클래스의 **범용성**을 위해서입니다. 단일 좀비 객체를 생성할 때는 `setName`을 별도로 부르는 것보다 생성과 동시에 이름을 부여하는 것이 더 효율적이고 깔끔하기 때문입니다.

* **Reference (`std::string&`)**: 기존 변수에 대한 **별명(Alias)**입니다. 새로운 메모리 공간을 확보하지 않고 원본 변수를 그대로 가리킵니다.

### 💡 Deep Dive

#### **Q1. 왜 원본 변수, 포인터가 가리키는 주소, 참조자의 주소가 모두 같나요?**
* **답변**: 포인터 변수(`stringPTR`)는 원본의 **주소값**을 값으로 들고 있기 때문이고, 참조자(`stringREF`)는 원본 변수 그 자체에 대한 **또 다른 이름**이기 때문입니다. 따라서 `&stringREF`를 하면 원본 변수인 `str`의 주소가 출력되는 것입니다.

#### **Q2. 포인터와 참조자의 가장 큰 차이점은 무엇인가요?**
* **답변**:
	* **포인터**: 주소를 저장하는 독립적인 변수이므로 `NULL`이 될 수 있고, 나중에 다른 주소를 가리키도록 변경이 가능합니다.
	* **참조자**: 선언과 동시에 **반드시 초기화**해야 하며, 한 번 초기화되면 평생 그 변수만 가리킵니다(재할당 불가). 또한 `NULL` 참조라는 개념이 존재하지 않아 더 안전합니다.

#### **Q3. 참조자를 사용하면 어떤 이점이 있나요?**
* **답변**: 포인터처럼 주소를 다루면서도, 문법적으로는 일반 변수처럼 사용할 수 있어 가독성이 좋습니다. 역참조 연산자(`*`)를 일일이 붙일 필요가 없으며, 포인터 연산으로 인한 실수나 `NULL` 접근 에러를 방지할 수 있습니다.

#### **Q4. 주소 출력 시 `&stringREF`와 `stringPTR`의 차이는?**
* **답변**: `stringPTR`은 그 변수가 담고 있는 **값(원본의 주소)**을 출력하는 것이고, `&stringREF`는 참조하고 있는 **원본 객체의 주소**를 직접 출력하는 것입니다. 결과적으로는 같지만 의미론적으로는 참조자가 원본과 '동일시'된다는 점을 보여줍니다.
- "코드를 보면 `stringPTR` 자체의 주소를 찍어보는 주석 처리된 부분이 있지? 그걸 풀어보면 `stringPTR`이라는 포인터 변수도 메모리 어딘가에 자기 자리를 갖고 있다는 걸 알 수 있어. 반면 참조자는 자기만의 주소가 따로 없다는 게 가장 큰 차이야!"

## 📂 Exercise 04: Sed is for losers
C++의 **파일 스트림(File Streams)**과 **문자열 조작**을 사용하여 기존의 `sed` 도구를 직접 구현해보며, 표준 라이브러리의 추상화 원리를 이해하는 단계입니다.

### Key Concepts
* **RAII (Resource Acquisition Is Initialization)**: 객체의 생명 주기를 통해 자원(파일 핸들러 등)을 자동으로 관리하여 자원 누수를 방지합니다.
* **Manual String Replacement**: `find()`, `substr()`, `append()`의 조합을 통해 치환 알고리즘을 구현합니다.
* **Stream Buffering**: `rdbuf()`와 `stringstream`을 사용하여 대량의 데이터를 효율적으로 처리합니다.

### 💡 Deep Dive: C++ 표준 라이브러리와 동작 원리

#### **Q1. 과제 이름이 왜 "Sed is for losers"인가요?**
* **답변**: 이미 만들어진 강력한 도구(`sed`)나 편리한 함수(`std::string::replace`)에만 의존하지 않고, 문자열 검색과 메모리 재할당, 스트림 제어 로직을 **직접 바닥부터 설계**해보라는 도전적인 의미가 담겨 있습니다.

#### **Q2. `std::string::replace` 없이 어떻게 치환을 구현했나요?**
* **답변**: `find()`로 원본에서 `s1`의 위치를 찾고, `substr()`로 검색 지점 이전까지의 내용을 잘라낸 뒤, `append()`를 사용하여 결과 문자열에 이전 내용과 `s2`를 차례로 이어 붙이는 루프를 구성했습니다. 더 이상 `s1`이 발견되지 않을 때(`npos`)까지 이 과정을 반복합니다.

#### **Q3. 파일 처리 시 C언어와 비교하여 C++의 장점은 무엇인가요?**
* **답변**: C언어는 `FILE*` 포인터를 수동으로 닫지 않으면 자원 누수가 발생하지만, C++의 `fstream` 객체는 **소멸자**가 자동으로 파일을 닫아줍니다(RAII). 또한, `std::string`이 메모리 크기를 스스로 조절하므로 버퍼 오버플로우 위험으로부터 안전합니다.

#### **Q4. `std::ifstream inputFile(filename.c_str());` 문법의 의미는?**
* **답변**: `std::ifstream` 클래스의 **객체**인 `inputFile`을 생성하면서, **생성자**의 인자로 파일 경로를 전달하여 객체 탄생과 동시에 파일을 열도록 명령하는 문장입니다.

#### **Q5. 왜 `filename.c_str()`을 사용해야 하나요?**
* **답변**: C++98 표준의 `ifstream` 생성자는 인자로 `std::string`을 직접 받지 못하고 C 스타일의 문자열(`const char *`)만 받을 수 있기 때문에, 통역사 역할을 하는 `c_str()` 멤버 함수로 변환해준 것입니다.

#### **Q6. 문자열(`std::string`)과 문자열 스트림(`std::stringstream`)의 차이는?**
* **답변**:
    * **`string`**: 데이터를 담고 있는 **물통**과 같으며, 특정 위치의 값에 접근하기 좋습니다.
    * **`stringstream`**: 데이터가 흐르는 **파이프**와 같으며, `<<` 연산자를 통해 다양한 타입의 데이터를 하나로 합치거나 `rdbuf()`를 이용해 파일 내용을 통째로 옮기는 버퍼 역할에 최적화되어 있습니다.

#### **Q7. `std::string::find()` 결과와 `std::string::npos`를 비교하는 이유는?**
* **답변**: `find()`가 일치하는 문자열을 찾지 못했을 때 반환하는 특수한 값이 `npos`입니다. 이 값을 체크하지 않고 인덱스로 사용하면 잘못된 메모리에 접근하게 되므로, 반드시 성공 여부를 확인한 뒤 다음 로직을 수행해야 합니다.

### 🛠️ 주요 사용 함수 및 객체 정리
* **`std::cerr`**: 에러 메시지 전용 출력 스트림.

* **`rdbuf()`**: 파일의 내부 스트림 버퍼에 직접 접근하여 효율적인 데이터 이동 가능.
* **`str()`**: 스트림에 쌓인 데이터를 `std::string` 형태로 추출.

* **Member Function Pointer**: 클래스에 속한 함수의 주소(정확히는 클래스 내 오프셋)를 저장하는 특수한 포인터 타입입니다.
* **Dispatch Table**: 특정 키(문자열)와 실행할 행위(함수 포인터)를 미리 연결해 둔 테이블을 통해 실행 시점에 즉각적으로 로직을 분기하는 설계 방식입니다.

### 💡 Deep Dive

#### **Q1. 과제 이름이 왜 "Harl 2.0"인가요?**
* **답변**: 원래는 'Karen'이라는 이름의 밈을 활용했으나, 보다 중립적인 표현을 위해 **Harl**로 변경되었습니다. **2.0**은 기존의 단순 조건문 기반 로직에서 멤버 함수 포인터를 사용하는 '진화된 형태'로 업데이트되었음을 상징합니다.

#### **Q2. `void (Harl::*funcs[])(void)` 문법의 상세 의미는 무엇인가요?**
* **답변**: `Harl` 클래스에 소속된 함수 중 **반환형이 `void`이고 매개변수가 없는(`void`) 함수**들을 가리키는 **포인터들의 배열**을 선언하는 문법입니다. 일반 함수 포인터와 달리 `Harl::*`를 통해 특정 클래스 영역(Scope)에 묶여 있음을 명시해야 합니다.

#### **Q3. 멤버 함수 포인터 호출 시 `(this->*funcs[i])()`처럼 복잡한 기호를 쓰는 이유는?**
* **답변**:
    1. **객체 결합**: 멤버 함수는 특정 주소만으로 실행될 수 없으며, 반드시 작업을 수행할 **객체(`this`)의 주소**가 필요하기 때문입니다.
    2. **우선순위 보호**: 함수 호출 연산자(`()`)가 멤버 포인터 연산자(`->*`)보다 우선순위가 높으므로, 반드시 괄호로 묶어 객체와 함수 포인터를 먼저 결합(`Binding`)한 뒤 호출해야 합니다.

#### **Q4. `if-else` 지옥 대신 멤버 함수 포인터를 쓰면 어떤 실무적 이점이 있나요?**
* **답변**:
    1. **유지보수성(OCP)**: 새로운 불평 레벨이 추가되어도 기존 `complain` 함수의 로직을 수정할 필요 없이 테이블(배열)에 한 줄만 추가하면 됩니다.
    2. **가독성**: 수많은 조건 분기문이 사라져 코드의 핵심 로직이 간결해집니다.
    3. **확장성**: 동적으로 실행할 함수를 교체해야 하는 콜백(Callback)이나 상태 패턴(State Pattern) 구현에 필수적입니다.

#### 팁
"문자열 배열(`levels[]`)과 함수 포인터 배열(`funcs[]`)의 인덱스를 일치시켜 루프를 돌리는 방식은 임베디드 시스템이나 게임 엔진의 명령 처리기에서 아주 흔히 쓰이는 패턴이야. 이 문법에 익숙해지면 C++의 강력한 다형성을 다룰 준비가 된 셈이지!"
