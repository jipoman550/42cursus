# 📖 C++ - Module 06

**C++ Casts (C++ 형 변환 연산자)**

## 📝 Introduction

이 프로젝트는 C++에서 데이터 타입을 안전하고 명확하게 변환하기 위한 4가지 핵심 형 변환 연산자(`static_cast`, `dynamic_cast`, `const_cast`, `reinterpret_cast`)의 개념과 올바른 사용법을 다룹니다. C 스타일 형 변환의 모호함과 위험성을 탈피하고, 컴파일 타임과 런타임에 안전하게 타입을 제어하는 방법을 학습합니다.

## 🛠️ General Rules

과제를 수행하며 반드시 준수해야 하는 규칙들입니다.

### 1. Compilation

* **Compiler**: `c++`
* **Flags**: `-Wall -Wextra -Werror`
* **Standard**: `-std=c++98` 플래그를 추가해도 완벽하게 컴파일되어야 합니다.

### 2. Forbidden & Allowed

* **C 함수 사용 금지**: `printf()`, `malloc()`, `free()` 사용 시 **0점** 처리됩니다.
* **사용 금지 키워드**: `using namespace`와 `friend` 사용 시 **-42점** 감점됩니다.
* **Orthodox Canonical Class Form (OCCF)**: 모든 클래스는 반드시 기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자를 포함해야 합니다.
* **C++ 스타일 캐스팅 강제**: C 스타일의 형 변환(예: `(int)variable`)은 일절 금지되며, 반드시 C++ 캐스팅 연산자를 사용해야 합니다.

## 🏛️ Exercise 00: Conversion of scalar types

가장 기본적이면서 원초적인 데이터 타입들(스칼라 타입: `char`, `int`, `float`, `double`)의 문자열 리터럴 표현을 판별하고, C++ 스타일의 형 변환 규칙에 맞춰 다른 모든 타입으로 안전하게 전파하는 프로그램을 설계합니다.

### Objectives

* 입력받은 문자열 리터럴의 원본 타입(`char`, `int`, `float`, `double`)을 정확히 감지.
* 감지된 원본 타입을 기준으로 다른 세 가지 타입으로의 명시적 형 변환(`static_cast`) 수행.
* 각 타입별 범위를 검증하여 출력 불가능(`Non displayable`) 또는 불가능(`impossible`) 여부를 판단하여 출력.

### 💡 Deep Dive

#### **Q1. `./convert -10`을 입력했을 때 `char: Non displayable`이 나오는 게 맞나요? 화면에 왜 아무것도 안 보이죠?**

* **표준 아스키의 한계**: C/C++의 `std::isprint()` 함수는 화면에 정상적으로 인쇄 가능한 문자(표준 아스키 `32 ~ 126`번)인지를 검사합니다.
* **확장/음수 영역의 특성**: 컴퓨터 내부에서 `-10`은 `unsigned char`로 해석하면 `246`번에 해당합니다. 이 영역은 터미널 인코딩 환경에 따라 글자가 깨지거나 출력할 수 없는 제어 문자 취급을 받기 때문에 `std::isprint()`가 거짓(False)을 반환합니다.
* **방어적 프로그래밍**: 터미널 환경에 따라 깨진 글자가 출력되는 위험을 원천 차단하기 위해, 범위를 벗어나는 문자는 눈에 보이지 않더라도 안전하게 `Non displayable`로 방어 처리하는 것이 출제자의 의도에 완벽히 부합합니다.

#### **Q2. `÷` (나누기 기호) 같은 특수 문자도 출력 가능해야 하는 것 아닌가요? 왜 `char`에서는 안 되죠?**

* **자료형의 크기 한계**: `char`는 정확히 1바이트(8비트) 크기의 상자입니다. 반면 현대 리눅스 환경(UTF-8)에서 `÷`나 한글 같은 특수 문자는 보통 2바이트 이상을 차지하는 거대한 데이터입니다.
* **데이터의 손실**: 거대한 문자 데이터를 1바이트 상자에 강제로 구겨 넣으면 비트가 잘려 나가는 데이터 왜곡이 발생합니다. 표준 함수 `std::isprint()`는 이러한 환경별 글자 깨짐을 막기 위해 7비트 기본 아스키 영역을 벗어나는 모든 확장 문자에 대해 무조건 거짓을 반환하므로, 안전하게 `Non displayable`로 필터링해야 합니다.

#### **Q3. `./convert 2147483649`를 넣었을 때 `int: impossible`이 나와야 하는데, 왜 오버플로우가 난 엉뚱한 음수 값이 출력될까요?**

* **오염된 재료의 재사용 문제**: 입력값이 `int` 범위(21억)를 넘었을 때, 원본 `long l_val`에서 `i_val = static_cast<int>(l_val);`을 실행하면 `int` 상자가 넘쳐 변수 내부가 이미 `-2147483647`이라는 쓰레기 값으로 오염됩니다. 이 상태에서 이미 깨진 `i_val`을 가지고 `double d`나 `float f`를 만들면 전염병처럼 모든 실수가 음수로 오염됩니다.
* **해결책 (독립적인 캐스팅)**: 이미 터져버린 `i_val`을 다음 캐스팅의 재료로 쓰지 말고, 온전한 원본 값이 살아있는 `long l_val`을 끝까지 베이스캠프로 삼아 `f = static_cast<float>(l_val);`, `d = static_cast<double>(l_val);`과 같이 각각 독립적으로 캐스팅해야 합니다. 그래야 터질 상자만 터지고, 살 수 있는 거대한 실수 상자들은 원본 수치를 안전하게 보존할 수 있습니다.

#### **Q4. 정수(`int`, `char`) 출력부와 달리, `float`와 `double` 출력부에는 왜 `ref < min || ref > max` 같은 복잡한 범위 검사문이 없나요?**

* **스스로 상태를 바꾸는 실수**: 정수 타입은 상자가 넘치면 비트가 뒤집혀 수치가 완전히 오염되지만, `float`와 `double` 같은 부동소수점 자료형은 표현 한계를 초과하면 IEEE 754 표준에 따라 스스로 `inf` (무한대)라는 특수한 상태 값으로 변환됩니다.
* **과제 가이드라인 준수**: 과제 PDF에서도 실수 범위의 오버플로우는 `impossible`이 아니라 `inf`나 `inff`로 표현하라고 요구합니다. 실수가 가질 수 있는 최대 한계치($10^{308}$)를 완전히 초과하는 심각한 에러는 이미 위쪽 파싱 단계에서 `isImpossible` 플래그가 사전에 필터링해 주므로, 하단 출력부에서는 포맷 제어만 신경 쓰면 됩니다.

#### **Q5. 소수점이 없는 정수 형태의 실수를 출력할 때 `std::fixed`와 `std::setprecision(1)`을 적용하는 특별한 이유가 있나요?**

* **스트림 엔진의 기본 성질**: C++의 `std::cout`은 실수를 출력할 때 소수점 아래에 의미 없는 `0`만 존재하면 가독성을 위해 소수점을 생략해 버립니다 (예: `42.0` ➔ `42`).
* **출력 가이드라인 맞춤**: 과제 PDF의 출력 예시를 보면 정수 `0`을 넣었을 때 `float: 0.0f`, `double: 0.0`과 같이 소수점 첫째 자리를 유지하고 있습니다. 따라서 `std::fixed`(고정 소수점 표기)와 `std::setprecision(1)`을 결합하여, 소수점이 없는 깨끗한 정수 형태의 실수일지라도 강제로 뒤에 `.0`을 인쇄하도록 규격을 맞춰주는 디펜스 코드가 필수적입니다.

#### **Q6. `./convert 4.23410013123123...` 처럼 소수점을 아주 길게 주었는데, 왜 `float`와 `double` 결과가 둘 다 `4.2341`로 뭉개져서 똑같이 나올까요?**

* **기본 유효숫자 한계**: `std::cout`은 실수를 화면에 그릴 때 아무런 설정을 주지 않으면 **앞에서부터 딱 6자리의 유효숫자만 출력**하는 표준 규칙을 가집니다.
* **동일한 출력의 비밀**: 입력한 문자열이 소수점이 살아있는 실수라면 포맷팅을 타지 않고 `else` 블록으로 떨어집니다. 조작자가 없는 `std::cout` 엔진에 의해 앞의 6자리인 `4.23410`만 채택되고 맨 뒤의 무의미한 `0`이 지워지면서 세 케이스 모두 눈에는 똑같이 잘린 모양으로 보이게 됩니다. (실제 변수 내부 메모리 상자에는 데이터가 최대한 정밀하게 살아있습니다.)

#### **Q7. `./convert 4294967295`를 넣었더니 `float` 값이 `4294967296.0f`로 끝자리가 1 늘어나서 출력됩니다. 버그인가요?**

* **자료형의 정밀도 한계**: `float`(4바이트) 상자는 아주 큰 숫자까지 담을 수는 있지만, 숫자를 완벽하게 기억할 수 있는 정밀도(가수부)는 **십진수 딱 7자리**에 불과합니다.
* **반올림 오차(Rounding Error)**: 입력된 `4294967295`는 10자리 숫자이므로 `float`의 용량을 초과합니다. 컴퓨터가 이진수로 변환하여 구겨 넣는 과정에서 비트가 잘려 나가고, `float`가 표현할 수 있는 가장 가깝고 오차가 적은 비트 수치인 `4294967296`($2^{32}$)으로 자동 반올림된 것입니다. 이는 C++ 하드웨어 규격상 지극히 정상적인 결과입니다.

#### **Q8. 미스터리입니다. `./convert -9223372036854775808`은 19자리나 되는 엄청나게 긴 거대 숫자인데, 왜 `float`와 `double`에 오차 없이 완벽하고 정교하게 출력되나요?**

* **2진수 저장 방식의 비밀**: 컴퓨터는 숫자를 10진수가 아니라 내부적으로 2진수 과학적 표기법($\text{가수부} \times 2^{\text{지수부}}$)으로 저장합니다.
* **치트키 구조**: 입력된 `-9223372036854775808`은 수학적으로 정확히 $-2^{63}$입니다. 이를 2진수 표기법으로 변환하면 단순히 $-1.0 \times 2^{63}$이 됩니다. 숫자를 기억하는 가수부 상자에는 단지 `1.0`이라는 너무나 단순한 수치만 들어가면 끝나기 때문에, `float` 상자의 용량이 아무리 좁아도 단 1비트의 오차도 없이 자릿수를 완벽하게 유지하며 표현할 수 있는 것입니다. (여기서 1을 더한 `-9223372036854775807`을 넣으면 비트 구조가 복잡해져 즉시 정밀도가 파괴되며 반올림 오차가 발생합니다.)

#### **Q9. `ScalarConverter.cpp`를 리팩토링하면서 도우미 함수들의 매개변수에서 `int i_val` 부분을 아예 지워버려도 과제 규칙에 위배되지 않나요?**

* **완전히 안전합니다**: 과제 PDF에서 요구하는 강제 규칙은 "문자열을 분석한 뒤 진짜 오리지널 타입을 기반으로 명시적 형 변환(`static_cast`)을 완벽히 수행하라"입니다. 우리는 이미 `convert` 내부의 `if-else` 파싱 단계에서 이를 완벽하게 마쳤습니다.
* **자유로운 설계 영역**: 메인 로직이 너무 길어져 내부적으로 분리한 `printChar`나 `getOriginalValueAsDouble` 같은 도우미 함수들은 우리가 임의로 설계한 편의용 도구일 뿐입니다. 사용하지 않는 인자인 `i_val`을 호출부와 매개변수 양쪽에서 완전히 도려내는 것이, 컴파일러 경고(`unused-parameter`)를 막고 메모리를 아끼는 훨씬 깔끔하고 정석적인 C++ 리팩토링 방식입니다.
