#### **Q1. `while (std::getline(dbFile, line))` 루프는 내부적으로 한 줄씩 어떻게 읽어 들이며, 파일 끝(EOF) 도달 시 조건문이 종료되는 원리는 무엇인가요?**

* **스트림 참조자 반환**: `std::getline` 함수는 `bool` 값이 아니라 인자로 받은 파일 스트림 객체(`dbFile`)의 참조자를 그대로 돌려줍니다. 즉, 함수 실행 직후 `while` 조건문 자리에는 스트림 객체 자체가 남게 됩니다.
* **`operator bool()`의 상태 판별**: C++ 컴파일러는 조건문 내부의 객체를 판별하기 위해 스트림 내부에 정의된 타입 변환 연산자를 호출합니다. 정상적으로 한 줄을 읽는 동안에는 내부 표시등이 `goodbit` 상태이므로 `true`로 자동 변환되어 루프가 돕니다. 반면, 파일 맨 마지막 줄까지 다 읽은 후 한 번 더 읽으려고 시도하면 데이터가 없으므로 내부 표시등 중 `eofbit`와 `failbit`가 켜지고, 컴파일러는 이를 `false`로 인식하여 루프를 안전하게 탈출시킵니다.

#### **Q2. 문자열을 파싱할 때 `int` 대신 굳이 `size_t commaPos`와 같이 `size_t` 타입을 사용하는 이유가 무엇인가요?**

* **표준 타입과의 일치**: `std::string::find` 멤버 함수가 반환하는 인덱스의 실제 타입이 바로 `size_t`입니다. `size_t`는 부호 없는 정수형(unsigned)으로, 메모리나 객체가 가질 수 있는 최대 크기를 표현하도록 설계되었습니다.
* **오버플로우 및 부호 비트 방지**: `int` 타입은 부호가 있고 4바이트로 고정되어 있어, 기가바이트(GB) 단위의 거대한 문자열을 파싱할 때 표현 범위를 초과하는 오버플로우가 날 수 있습니다. 또한, 찾기 실패를 뜻하는 `std::string::npos`와 비교할 때 부호 비트 불일치로 인한 예기치 못한 버그를 유발할 수 있으므로 C++ 표준 관례에 맞춰 `size_t`를 사용하는 것이 정석입니다.

#### **Q3. 문자열 검증 및 맵 검색 전에 직접 구현한 `trim` 멤버 함수를 거쳐야 하는 필수적인 전처리 이유는 무엇인가요?**

* **매칭 실패 차단**: 외부 파일 데이터는 `2011-01-03 | 3`과 같이 기호 앞뒤로 공백을 포함하는 경우가 많습니다. `trim`을 통해 문자열 양 끝의 공백(` \t\n\r\f\v`)을 완벽히 제거하지 않으면, 날짜 문자열 뒤에 공백이 숨어들어 `"2011-01-03 "` 형태가 됩니다. 이는 데이터베이스 내부의 `"2011-01-03"`과 완전히 다른 문자열로 취급되어 `std::map` 검색 시 데이터를 찾지 못하는 대참사가 일어납니다.
* **수치 변환 예외 방지**: 값(Value) 문자열 역시 앞에 공백이 잔존해 있으면 향후 `std::strtod` 변환 루프나 엄격한 포맷 검증 단계에서 예기치 못한 차단(Error)이 발생할 수 있으므로 공백을 깎아내어 데이터의 정밀도를 순수하게 유지해야 합니다.

#### **Q4. 날짜 유효성 검사(`validateDate`) 내부에서 비트코인 창시년도 이전(`year < 2009`)을 철저히 입구 컷 하고 있음에도, 환율 검색 후 `if (rate < 0)` 예외 처리가 최종 수비수로서 반드시 자리를 지켜야 하는 상황은 무엇인가요?**

* **2009년 극초반의 데이터 공백**: 비트코인의 제네시스 블록은 2009년 1월 3일에 생성되었기 때문에, `data.csv` 데이터베이스는 대개 `2009-01-02` 혹은 `2009-01-03`부터 시작합니다. 만약 사용자가 달력상 정상 날짜인 `2009-01-01`을 입력하면 `validateDate`는 연도가 2009년이므로 무사 통과시킵니다.
* **데이터베이스의 한계점 방어**: 그러나 `findRate` 함수가 내부 탐색을 시작하면 `2009-01-01`보다 과거 데이터가 DB에 없으므로 매칭할 환율이 존재하지 않아 `-1.0f`를 리턴하게 됩니다. 이처럼 **달력상으로는 멀쩍하나 우리 데이터베이스가 가진 정보의 한계점보다 과거의 날짜가 유입되는 상황** 및 **DB 파일이 통째로 비어 있는 극한의 상황**에서 돈이 음수로 계산되는 치명적인 버그를 막기 위해 `if (rate < 0)` 검사는 반드시 필요합니다.

#### **Q5. 문자열을 숫자로 판별하거나 변환할 때, `static_cast<unsigned char>`로 문자를 감싸서 `isdigit`나 `isspace`에 전달해야 하는 C++ 표준 구조적 이유는 무엇인가요?**

* **음수 인덱스 참조(Undefined Behavior) 방지**: 옛 C 라이브러리 유산인 `std::isdigit`나 `std::isspace` 함수는 내부적으로 매번 문자를 계산하는 오버헤드를 줄이기 위해 256개짜리 판별 배열(Lookup Table)을 만들어 두고 인덱스로 접근합니다.
* **확장 아스키 및 한글 차단**: C++에서 일반 `char`는 부호가 있는 `signed char`인 경우가 많아, 한글이나 깨진 바이너리, 아스키 범위를 벗어나는 확장 문자가 들어오면 메모리상에서 음수 값(예: `-64`)으로 표현됩니다. 형변환 없이 음수가 전달되면 함수 내부에서 `Table[-64]`와 같은 음수 인덱스 참조가 일어나며 세그폴트 크래시가 터지거나 프로그램이 정의되지 않은 상태로 뻗어버립니다. 이를 **`unsigned char`로 형변환 해주면 메모리 비트의 왜곡 없이 0~255 사이의 안전한 양수 값으로 변환**되므로, 크래시 없이 안전하게 `false`를 판별해 내어 프로그램을 완벽하게 보호할 수 있습니다.

#### **Q6. `std::strtod` 함수 변환 직후 `if (endptr == valueStr.c_str())` 조건문이 성립하는 원리와 이를 통해 검출해 내는 에러는 무엇인가요?**

* **포인터 위치의 대조**: `valueStr.c_str()`은 검사할 문자열의 맨 첫 번째 글자가 저장된 메모리 주소입니다. 반면 `std::strtod`는 문자를 읽다가 숫자가 아닌 글자를 만나는 순간 읽기를 포기하고 그 지점의 주소를 `endptr`에 심어줍니다.
* **단 1글자도 읽지 못한 상태 판정**: 만약 두 주소값이 완벽하게 일치한다면(`endptr == c_str()`), 이는 `strtod`가 첫 번째 문자부터 숫자가 아니라고 판단하여 **단 한 칸도 전진하지 못하고 즉시 파싱을 포기했음**을 의미합니다. 즉, 사용자가 숫자가 아닌 `"abc"`, `"hello"` 같은 100% 순수 문자열을 값 자리에 밀어 넣었을 때 이를 저수준 주소 비교 연산으로 가장 빠르고 정밀하게 포착해 내는 격리 장치입니다.

#### **Q7. 값이 너무 길어 `double` 그릇마저 터져버리는 초거대 숫자(예: 9가 100개 연속된 수)가 유입될 때, `Error: too large a number.` 가 정상 출력되는 원리는 무엇인가요?**

* **`HUGE_VAL` (무한대) 전환**: `std::strtod`는 숫자가 아무리 길어도 중간에 파싱을 포기하지 않고 널 문자(`\0`)까지 끝까지 읽어 나갑니다. 다만 `double` 타입의 최대 표현 범위($1.79 \times 10^{308}$)를 초과했음을 감지하면, 변수 값을 컴퓨터 수학상의 무한대를 뜻하는 상수인 **`HUGE_VAL` (`inf`)**로 채워버립니다.
* **무한대 대소 비교 연산**: 이후 코드는 범위 검사 식인 `if (value > 1000)` 문장을 만나게 됩니다. 컴퓨터 내부 연산에서 무한대(`inf`)는 당연히 `1000`보다 큰 수로 판정(`true`)되므로, 데이터가 오염되거나 `bad input`으로 오작동하는 일 없이 서브젝트가 요구하는 정확한 가이드라인인 **`Error: too large a number.`** 에러 궤도로 완벽하게 안착하여 처리됩니다.

#### **Q8. `std::atoi` 함수는 예외 처리가 불가능한 결함이 있는데, 본 로직의 연/월/일 파싱 부분에서 어떻게 안전하게 사용될 수 있었나요?**

* **상위 로직의 철저한 사전 검사**: `std::atoi`는 잘못된 문자열이 들어오면 오류를 감지하지 못하고 엉뚱하게 `0`을 뱉거나 오버플로우 시 쓰레기 값을 반환하는 약점이 있습니다.
* **완벽한 환경 조성**: 그러나 현재 코드는 `atoi`를 호출하기 전, 1단계 필터(`date.length() != 10`)를 통해 문자열 길이를 완벽히 제어하고, 2단계 필터(`std::isdigit` 루프)를 통해 날짜 포맷 내에 숫자가 아닌 모든 문자 예외 입력을 사전에 철저히 입구 컷 하고 있습니다. 결과적으로 `atoi` 내부로는 오직 검증된 범위의 순수 숫자 문자열 조각(`"2026"`, `"06"`)만 도달하므로, 함수 본연의 한계를 상위 계층의 다중 방어벽으로 완벽히 통제하여 절대적인 안정성을 보장합니다.

#### **Q9. 가장 가까운 과거 날짜를 찾기 위해 사용한 `lower_bound` 방식 대신 `std::map::upper_bound`만 사용하는 대안을 고려했을 때의 구조적 차이와 장단점은 무엇인가요?**

* **`upper_bound` 대안의 간결성**: `std::map::upper_bound(date)`는 입력 날짜보다 무조건 큰(`>`) 첫 번째 원소를 찾습니다. 일치하는 날짜가 있든 없든 상관없이, `upper_bound`가 반환한 반복자를 한 칸 뒤로 당기면(`--it`) 우리가 원하는 '당일 혹은 가장 가까운 과거 날짜'를 가리키게 됩니다. 이 방식을 채택하면 코드 맨 위에 위치한 `_data.find(date)` 일치 검사 구문을 통째로 생략할 수 있어 소스 코드가 비약적으로 간결해집니다.
* **`lower_bound` 명시적 조합의 의의**: 그럼에도 현재의 `find` + `lower_bound` 조합은 "정확히 일치하는 날짜가 존재할 때는 반복자 감소 연산(`--it`)을 단 한 번도 수행하지 않고 즉시 안전하게 값을 반환한다"는 **명시적인 직관성과 경계선 예외 안전성(Exception Safety)**을 제공합니다. 두 방식 모두 시간 복잡도는 $O(\log N)$으로 성능이 동일하므로, 개발자의 설계 스타일과 안정성 가치관에 따른 정당한 선택지입니다.

#### **Q10. `findRate` 멤버 함수 내부에서 `std::map`을 탐색할 때 일반 `iterator`가 아닌 `const_iterator`를 강제해야만 하는 컴파일 구조적 이유는 무엇인가요?**

* **`const` 함수 계약의 이행**: `findRate` 함수는 클래스의 내부 멤버 변수 데이터를 수정하지 않고 조회만 하겠다는 약속으로 함수 선언부 우측에 **`const` 키워드**가 마킹되어 있습니다.
* **읽기 전용 상태 전이**: 함수가 `const` 상태가 되면 내부 멤버 변수인 맵 컨테이너(`_data`) 역시 자동으로 '읽기 전용(Read-Only)' 상태로 엄격히 통제됩니다. 만약 내부 원소의 수정을 허용하는 일반 `iterator`로 맵을 탐색하려고 시도하면 컴파일러는 객체의 상수성 계약이 깨질 위험을 감지하고 즉시 컴파일 에러를 발생시킵니다. 따라서 데이터를 절대 오염시키지 않는 안전한 전용 나침반 포인터인 **`const_iterator`**를 사용하는 것이 C++ 정적 타입 시스템 명세에 부합하는 정석적인 설계입니다.

## 🏛️ Exercise 01: Reverse Polish Notation

주어진 역폴란드 표기법(후위 표기법, Postfix Notation) 수식 문자열을 정밀하게 분석하고, 스택(Stack) 자료구조의 후입선출(LIFO) 특성을 활용하여 사칙연산을 정확하게 수행하는 연산기 프로그램을 설계합니다.

### Objectives

* 전 단계(`ex00`)에서 소모한 `std::map` 컨테이너 카드를 제외하고, 과제의 요구 사양에 완벽히 부합하는 최적의 STL 컨테이너인 `std::stack` 아키텍처의 원리 및 한계 파악.
* 문자열 스트림(`std::stringstream`) 토큰 분리 메커니즘을 적용하여 불규칙한 공백 포맷을 유연하게 흡수하는 방어적 파싱 로직 구현.
* C++의 '값 복사 모델'과 타 언어의 '참조 모델' 간의 구조적 차이를 이해하고, 예외 발생 시 자원 유실을 원천 차단하는 '강한 예외 안전성(Strong Exception Guarantee)' 설계 철학 숙지.

### 💡 Deep Dive

#### **Q1. `std::stringstream`을 활용한 토큰 분리 방식의 메커니즘과, 연속된 다중 공백 포맷을 코딩 한 줄 없이 완벽하게 흡수할 수 있는 상세 원리는 무엇인가요?**

* **스트림 데이터 통로의 형성**: `std::stringstream ss(expression);` 구문은 인자로 전달받은 문자열을 하나의 물리적인 파일 스트림이나 입력 스트림(`std::cin`)처럼 취급할 수 있도록 메모리 버퍼 통로를 개설합니다.
* **추출 연산자의 자동 스킵 기능**: `while (ss >> token)` 루프에서 사용되는 추출 연산자(`>>`)는 스트림 내부 데이터를 읽을 때 스페이스, 탭, 뉴라인(`\n`) 등 모든 공백 문자(Whitespace)를 기준으로 데이터를 자동 파싱합니다. 이 과정에서 공백이 한 칸이든 수십 칸이든 관계없이 연속된 공백 문자열 덩어리를 컴파일러가 알아서 무시하고 다음 알맹이 문자열만 추출하므로, C 스타일의 `strtok`처럼 원본 문자열을 파괴하거나 복잡한 전처리 조건문 분기 없이도 악의적인 다중 공백 입력 환경을 유연하게 방어해 낼 수 있습니다.

#### **Q2. 피연산자 문자 조건 검사(`token.length() == 1 && std::isdigit`)를 통해 서브젝트의 '10 미만 정수' 제한을 철저히 제어하면서, 음수 기호(`-`) 유입 시 오작동을 완벽하게 걸러내는 파싱 필터의 원리는 무엇인가요?**

* **한 자릿수 제약의 명시적 차단**: 서브젝트는 입력 단계에서의 피연산자를 무조건 10 미만의 양의 정수(0~9)로 강제하고 있습니다. 이에 따라 토큰의 길이를 검사하는 `token.length() == 1` 장벽을 가장 먼저 세워둠으로써, `"10"`, `"42"` 같은 두 자릿수 이상의 정수 유입을 원천 차단하여 `RPNException`으로 유도합니다.
* **음수 포맷의 자동 격리**: 만약 사용자가 수식 중간에 음수를 표현하기 위해 `"-5"` 같은 토큰을 입력하면, 이 역시 길이가 2이기 때문에 숫자로 처리되지 않고 최종 `else` 분기로 떨어져 에러 처리가 수행됩니다. 또한 단순한 빼기 연산자 기호(`"-"`)가 단독 토큰으로 들어오면, 길이는 1이지만 `isdigit` 검사에서 `false`를 반환하므로 숫자가 아닌 정상적인 '연산자 판단 분기(`op == '-'`)'로 유도되어 파싱 시스템의 완벽한 교통정리가 완성됩니다.

#### **Q3. 스택(Stack)의 후입선출(LIFO) 아키텍처 관점에서 연산자 토큰을 만났을 때 먼저 꺼낸 값을 `rhs`(우항), 나중에 꺼낸 값을 `lhs`(좌항)로 분리 적재하여 연산해야 하는 수학적 정당성은 무엇인가요?**

* **메모리 적재 순서의 역전 현상**: 예를 들어 후위 표기법 수식 `"3 4 -"`는 인간의 수학식으로 변환하면 `3 - 4`가 되어야 합니다. 이를 컴퓨터가 탐색할 때는 순서대로 `3`을 먼저 넣고 그 위에 `4`를 쌓아 올리게 됩니다.
* **우항과 좌항의 정확한 조준**: 연산자 기호 `"-"`를 만나는 순간, 스택의 맨 위(`top()`)에서 가장 마지막에 적재되었던 값인 `4`가 가장 먼저 뽑혀 나오게 됩니다. 따라서 먼저 나온 데이터를 연산자 기호의 우측 항을 담당하는 **`rhs` (Right-Hand Side)** 변수에 담고, 그다음 한 칸 내려가 파묻혀 있던 `3`을 꺼내어 좌측 항을 담당하는 **`lhs` (Left-Hand Side)** 변수에 배치해야만, 부호가 민감한 뺄셈이나 나눗셈 연산 시 `3 - 4 = -1`이라는 수학적으로 완벽하고 정밀한 연산 결과값을 유도해 낼 수 있습니다.

#### **Q4. 0으로 나누기 (`Division by Zero`) 예외 상황을 소프트웨어 레벨(`if (rhs == 0)`)에서 명시적으로 제어하지 않았을 때, 운영체제(OS)와 런타임 단계에서 발생하는 시스템 대참사는 무엇인가요?**

* **하드웨어 인터럽트의 발동**: CPU 아키텍처 레벨에서 정수 연산 중 나눗셈의 분모가 `0`이 되는 상황은 정상적인 ALU(연산 장치) 제어가 불가능한 결함 상황입니다. 시스템 소스 레벨에서 이를 우회하지 않고 `lhs / rhs` 연산을 그대로 실행하면 CPU는 즉시 하드웨어 인터럽트를 발생시킵니다.
* **코어 덤프와 강제 종료**: 이 하드웨어 신호를 수신한 운영체제는 해당 프로그램에게 부동 소수점 결함 시그널인 `SIGFPE (Floating Point Exception)`를 전송합니다. 이 시그널을 받은 프로그램은 어떠한 C++ 내부 예외 복구 구문(`try-catch`)으로도 수습하지 못하고, 그 자리에서 즉시 메모리 코어 덤프를 남기며 프로세스가 비정상 강제 크래시 처리됩니다. 소프트웨어의 완벽한 런타임 안전성을 지키기 위해서는 연산 실행 직전 반드시 명시적인 조건문 분기로 분모의 `0` 유입 여부를 선제 검사하여 안전하게 처리해야 합니다.

#### **Q5. `_stack`과 `executeOperation`을 `private` 영역에 격리하고 `calculate`만 `public`으로 개방한 캡슐화(Encapsulation) 설계가 객체의 무결성과 유지보수성에 기여하는 아키텍처적 이점은 무엇인가요?**

* **불법적인 데이터 오염 통제**: 만약 핵심 적재 그릇인 `_stack` 변수가 외부 세상에 `public`으로 개방되어 있다면, 외부 소스코드(`main.cpp` 등)에서 클래스의 통제를 우회하여 10 이상의 숫자를 직접 강제로 `push`하거나 멀쩡히 계산 중인 원소를 멋대로 `pop`하여 지워버리는 등 데이터 정합성이 순식간에 붕괴할 위험이 있습니다.
* **구현 상세 은닉을 통한 느슨한 결합**: 외부 사용자에게는 오직 문자열을 던지면 정답이 나온다는 단순명료한 **`public` 인터페이스(`calculate`)**만 제공하고, 내부적인 스택 원소 추출 및 사칙연산 분기 기법(`executeOperation`)은 완전히 베일에 가려둠으로써 최소 권한의 원칙을 달성합니다. 이로 인해 향후 사칙연산 처리 로직을 대대적으로 리팩토링하거나 데이터 형식을 변경하더라도 외부 호출 코드를 단 한 줄도 건드리지 않고 클래스 내부만 수정하면 되므로 코드의 유지보수 효율성이 극대화됩니다.

#### **Q6. `<stdexcept>` 헤더를 포함하여 커스텀 예외 클래스(`RPNException`)를 디자인할 때, `std::exception` 상속 및 C++98 명세인 `throw()` 기표를 가상 함수 `what()` 뒤에 명시해야 하는 예외 안전성 구조는 무엇인가요?**

* **다형성 기반의 일괄 포착 기반 마련**: C++ 표준 예외의 근간인 `std::exception`을 복사 상속받음으로써, `RPNException` 객체는 표준 규격을 따르는 공식 예외 인스턴스로 공인됩니다. 이 덕분에 호출부인 `main.cpp` 계층에서는 구체적인 자식 클래스 타입을 미시적으로 알 필요 없이 **부모 클래스 참조자 자료형 상자(`const std::exception& e`)** 하나만 쳐두면 시스템 내에서 발생하는 모든 커스텀 예외들을 누수 없이 일괄적으로 안전하게 낚아챌 수 있습니다.
* **예외 지정자(Exception Specification)의 계약 보호**: 가상 함수 재정의부 맨 뒤에 붙은 `throw()` 키워드는 **"이 `what()` 함수가 동작하는 동안에는 내부 메모리 오버헤드 등 그 어떤 부수적인 예외도 절대로 발생시켜 외부로 다시 던지지 않겠다"**라고 컴파일러에게 엄격한 안전 서약을 선언하는 문법입니다. 에러 문장을 출력하려다가 시스템 내부에서 2차 예외가 발생하여 프로그램이 예외 복구 불능 상태로 완전 파멸(Crash)하는 현상을 완벽하게 봉쇄하는 C++ 표준 가이드라인입니다.

#### **Q7. 다른 언어(Java 등)의 스택과 달리 C++ STL `std::stack::pop()` 함수가 값을 반환하지 않고 `void`형으로 설계된 근본적인 메모리 모델(값 복사 vs 참조)과 '강한 예외 안전성(Strong Exception Guarantee)'의 관계는 무엇인가요?**

* **자바의 참조 모델 안전성**: 자바는 모든 인스턴스를 힙(Heap) 영역에 생성하고 변수에는 오직 주소값만 담는 참조 모델을 사용합니다. `pop()` 실행 시 이미 존재하는 객체의 가벼운 주소 숫자(32/64비트)만 리턴하므로 메모리 부족 등의 오류가 발생할 여지가 전혀 없어 값을 반환하면서 원소를 지워도 완벽히 안전합니다.
* **C++의 값 복사 위험성과 데이터 유실**: 반면 C++은 객체의 알맹이 그 자체를 전달하는 값 모델을 지원하므로, `pop()`이 값을 반환하는 구조라면 반환 과정에서 원본 데이터를 외부 변수에 복제하기 위해 **'복사 생성자'**를 실행하게 됩니다. 만약 스택 내부에서 맨 위 원소를 완전히 제거한 직후, 반환을 위해 복사 생성자가 호출되는 타이밍에 힙 메모리가 부족하여 **`std::bad_alloc` 예외가 발생**한다면 호출자는 데이터 복사본을 받지 못했는데 스택 내 원본은 이미 소멸해 버린 **"데이터 영구 공중분해 대참사"**가 터집니다.
* **원자성 보장을 위한 책임을 분리**: C++ 표준 위원회는 "실패 시 함수 호출 전 상태로 완벽히 복원되어야 한다"는 **강한 예외 안전성**을 충족하기 위해, 단순 조회를 담당하여 예외 발생 시에도 원본 데이터 손실 위험이 없는 `top()` 함수와, 자원 제거 역할만 안전하게 완벽 완수하는 `pop() (void형)` 함수로 인터페이스의 역할을 철저히 분리(Single Responsibility)시켰습니다.

#### **Q8. `new` 키워드를 직접 사용하지 않았음에도 `std::stack` 내부의 정수 데이터들이 실제로 머무는 물리적 메모리 구역(Stack Area vs Heap Area)은 어디이며, 프로그램 종료 시 누수 없이 해제되는 RAII 메커니즘은 어떻게 작동하나요?**

* **두 메모리 영역의 기하학적 공존**: `main.cpp` 소스 레벨에서 `RPN rpn;` 형태로 선언된 인스턴스 틀 자체는 함수의 지역 범위(Scope) 안에 종속되므로 컴파일러에 의해 관리되는 자동 변수 구역인 **메모리 스택 영역(Stack Area)**에 자리를 잡습니다. 그러나 우리가 숫자를 파싱하여 `_stack.push(value);`를 호출할 때마다, 내부의 실제 정수 알맹이 데이터 공간은 스택이 아닌 광활한 물리 공간인 **힙 영역(Heap Area)**에 수시로 메모리를 요청하여 동적 할당 처리됩니다.
* **소멸자 체인과 자원의 자동 회수**: 비록 개발자가 수동으로 `delete`를 적지 않았더라도 C++의 핵심 이념인 **RAII(Resource Acquisition Is Initialization)** 시스템이 작동합니다. `main` 함수가 끝나거나 스코프를 탈출하여 수명을 다한 `rpn` 객체가 메모리 스택 영역에서 소멸할 때, 클래스의 소멸자 `~RPN()`이 호출되면서 멤버 변수인 `_stack` 객체의 소멸자도 연쇄적으로 자동 격화됩니다. `std::stack` 소멸자는 내부 힙 공간에 조각조각 적재해 두었던 모든 정수 노드들의 주소를 루프 돌며 알아서 완벽하게 청소(`delete`)해 주는 자원 책임 관리 책무를 수행하므로, 스택에 최종 잔여물이 몇 개가 남았든 단 1바이트의 메모리 누수(Memory Leak)도 없이 완벽하게 방어가 완료됩니다.

#### **Q9. `std::stack`이 독립적인 컨테이너가 아닌 '컨테이너 어댑터(Container Adapter)'로 분류되는 이유와, 기본 알맹이 카드로 `std::vector`나 `std::list` 대신 `std::deque`를 채택하여 얻는 성능 및 캐시 적중률 최적화의 원리는 무엇인가요?**

* **변형된 인터페이스 포장지**: `std::stack`은 스스로 데이터를 관리하는 저수준 메모리 선언 로직이 없습니다. 기존의 순차 STL 컨테이너를 내부에 멤버 변수로 소유한 뒤, 오직 후입선출(RPN의 논리 구조) 기능만 수행하도록 나머지 불필요한 인터페이스들을 전부 가위질하여 차단한 **'기능 제한형 포장지'**이기 때문에 어댑터로 명명됩니다.
* **`vector` 대비 연속 비용의 감쇄**: 템플릿 선언 시 컨테이너 카드를 명시하지 않으면 기본 내장 카드로 **`std::deque`**가 강제 매칭됩니다. `std::vector`는 공간 포화 시 메모리를 통째로 이사하는 거대한 재할당 및 복사 오버헤드 시한폭탄이 동반되는 반면, `deque`는 메모리를 일정 블록 단위로 분할 관리하므로 데이터가 아무리 누적 삽입되어도 고정된 상수 시간 **$O(1)$의 안정적인 푸시 성능**을 뽑아냅니다.
* **`list` 대비 캐시 효율의 극대화**: `std::list`는 노드 단위로 데이터가 온 사방의 힙 메모리로 파편화(Fragmentation)되어 할당되므로 포인터 주소를 타고 들어갈 때 캐시 미스(Cache Miss) 확률이 폭증합니다. 반면 `deque`는 개별 블록 내부의 메모리가 완벽히 연속적인 행렬 구조를 이루고 있으므로 하드웨어 단위의 **캐시 적중률(Cache Locality)과 메모리 이용 효율성이 리스트 대비 압도적으로 정밀하고 조밀하게 최적화**됩니다.

#### **Q10. 수식 처리가 모두 끝난 시점에 `_stack.size() != 1` 조건을 정밀 검사함으로써 사전에 탐지해 내는 RPN 수식 오류 상황은 구체적으로 무엇인가요?**

* **피연산자 과다 적재 결함 추적**: 올바르게 작성된 역폴란드 표기법 수식은 연산자를 만날 때마다 숫자가 2개씩 소모되고 결과가 1개씩 복귀하므로, 문장의 맨 끝 널 문자(`\0`) 영역에 도달했을 때 스택 안에는 오직 단 1개의 최종 결과물만 당당하게 군림해야 정상입니다.
* **불완전 수식의 발각 시점**: 만약 평가자가 연산 기호가 턱없이 부족한 불완전 수식(예: `"5 5 5 +"`)을 입력한다면, 수식 문자열을 파싱하며 `while` 루프가 도는 동안에는 피연산자 한 자릿수 규격이나 2개 이상 존재 유무 검사가 모두 아슬아슬하게 패스되어 내부 계산이 조용히 누적 진행됩니다. 그러나 루프가 종결된 직후 최종 관문인 `if (_stack.size() != 1)` 감시망에 걸려드는 순간, 스택 내부에 쓰이지 않고 붕 떠버린 잔여 피연산자 찌꺼기들(예: 원소 개수 2개 이상 잔존)의 정체가 확연히 탄로나면서, 정상 수식이 아님을 완벽히 간파해 내어 표준 에러 가이드라인인 **`Error`** 궤도로 완벽하게 인입 처리할 수 있게 됩니다.
