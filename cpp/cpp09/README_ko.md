## 📝 Introduction

### 💡 Deep Dive

#### **Q1. `while (std::getline(dbFile, line))` 루프는 내부적으로 한 줄씩 어떻게 읽어 들이며, 파일 끝(EOF) 도달 시 조건문이 종료되는 원리는 무엇인가요?**

* **스트림 참조자 반환**: `std::getline` 함수는 `bool` 값이 아니라 인자로 받은 파일 스트림 객체(`dbFile`)의 참조자를 그대로 돌려줍니다. 즉, 함수 실행 직후 `while` 조건문 자리에는 스트림 객체 자체가 남게 됩니다.
* **`operator bool()`의 상태 판별**: C++ 컴파일러는 조건문 내부의 객체를 판별하기 위해 스트림 내부에 정의된 타입 변환 연산자를 호출합니다. 정상적으로 한 줄을 읽는 동안에는 내부 표시등이 `goodbit` 상태이므로 `true`로 자동 변환되어 루프가 돕니다. 반면, 파일 맨 마지막 줄까지 다 읽은 후 한 번 더 읽으려고 시도하면 데이터가 없으므로 내부 표시등 중 `eofbit`와 `failbit`가 켜지고, 컴파일러는 이를 `false`로 인식하여 루프를 안전하게 탈출시킵니다.

#### **Q2. 문자열을 파싱할 때 `int` 대신 굳이 `size_t commaPos`와 같이 `size_t` 타입을 사용하는 이유가 무엇인가요?**

* **표준 타입과의 일치**: `std::string::find` 멤버 함수가 반환하는 인덱스의 실제 타입이 바로 `size_t`입니다. `size_t`는 부호 없는 정수형(unsigned)으로, 메모리나 객체가 가질 수 있는 최대 크기를 표현하도록 설계되었습니다.
* **오버플로우 및 부호 비트 방지**: `int` 타입은 부호가 있고 4바이트로 고정되어 있어, 기가바이트(GB) 단위의 거대한 문자열을 파싱할 때 표현 범위를 초과하는 오버플로우가 날 수 있습니다. 또한, 찾기 실패를 뜻하는 `std::string::npos`와 비교할 때 부호 비트 불일치로 인한 예기치 못한 버그를 유발할 수 있으므로 C++ 표준 관례에 맞춰 `size_t`를 사용하는 것이 정석입니다.

#### **Q3. 문자열 검증 및 맵 검색 전에 직접 구현한 `trim` 멤버 함수를 거쳐야 하는 필수적인 전처리 이유는 무엇인가요?**

* **매칭 실패 차단**: 외부 파일 데이터는 `2011-01-03 | 3`과 같이 기호 앞뒤로 공백을 포함하는 경우가 많습니다. `trim`을 통해 문자열 양 끝의 공백(` \t\n\r\f\v`)을 완벽히 제거하지 않으면, 날짜 문자열 뒤에 공백이 숨어들어 `"2011-01-03 "` 형태가 됩니다. 이는 데이터베이스 내부의 `"2011-01-03"`과 완전히 다른 문자열로 취급되어 `std::map` 검색 시 데이터를 찾지 못하는 대참사가 일어납니다.
* **수치 변환 예외 방지**: 값(Value) 문자열 역시 앞에 공백이 잔존해 있으면 향후 `std::strtod` 변환 루프나 엄격한 포맷 검증 단계에서 예기치 못한 차단(Error)이 발생할 수 있으므로 공백을 깎아내어 데이터의 정밀도를 순수하게 유지해야 합니다.

#### **Q4. 날짜 유효성 검사(`validateDate`) 내부에서 비트코인 창시년도 이전(`year < 2009`)을 철저히 입구 컷 하고 있음에도, 환율 검색 후 `if (rate < 0)` 예외 처리가 최종 수비수로서 반드시 자리를 지켜야 하는 상황은 무엇인가요?**

* **2009년 극초반의 데이터 공백**: 비트코인의 제네시스 블록은 2009년 1월 3일에 생성되었기 때문에, `data.csv` 데이터베이스는 대개 `2009-01-02` 혹은 `2009-01-03`부터 시작합니다. 만약 사용자가 달력상 정상 날짜인 `2009-01-01`을 입력하면 `validateDate`는 연도가 2009년이므로 무사 통과시킵니다.
* **데이터베이스의 한계점 방어**: 그러나 `findRate` 함수가 내부 탐색을 시작하면 `2009-01-01`보다 과거 데이터가 DB에 없으므로 매칭할 환율이 존재하지 않아 `-1.0f`를 리턴하게 됩니다. 이처럼 **달력상으로는 멀쩍하나 우리 데이터베이스가 가진 정보의 한계점보다 과거의 날짜가 유입되는 상황** 및 **DB 파일이 통째로 비어 있는 극한의 상황**에서 돈이 음수로 계산되는 치명적인 버그를 막기 위해 `if (rate < 0)` 검사는 반드시 필요합니다.

#### **Q5. 문자열을 숫자로 판별하거나 변환할 때, `static_cast<unsigned char>`로 문자를 감싸서 `isdigit`나 `isspace`에 전달해야 하는 C++ 표준 구조적 이유는 무엇인가요?**

* **음수 인덱스 참조(Undefined Behavior) 방지**: 옛 C 라이브러리 유산인 `std::isdigit`나 `std::isspace` 함수는 내부적으로 매번 문자를 계산하는 오버헤드를 줄이기 위해 256개짜리 판별 배열(Lookup Table)을 만들어 두고 인덱스로 접근합니다.
* **확장 아스키 및 한글 차단**: C++에서 일반 `char`는 부호가 있는 `signed char`인 경우가 많아, 한글이나 깨진 바이너리, 아스키 범위를 벗어나는 확장 문자가 들어오면 메모리상에서 음수 값(예: `-64`)으로 표현됩니다. 형변환 없이 음수가 전달되면 함수 내부에서 `Table[-64]`와 같은 음수 인덱스 참조가 일어나며 세그폴트 크래시가 터지거나 프로그램이 정의되지 않은 상태로 뻗어버립니다. 이를 **`unsigned char`로 형변환 해주면 메모리 비트의 왜곡 없이 0~255 사이의 안전한 양수 값으로 변환**되므로, 크래시 없이 안전하게 `false`를 판별해 내어 프로그램을 완벽하게 보호할 수 있습니다.

#### **Q6. `std::strtod` 함수 변환 직후 `if (endptr == valueStr.c_str())` 조건문이 성립하는 원리와 이를 통해 검출해 내는 에러는 무엇인가요?**

* **포인터 위치의 대조**: `valueStr.c_str()`은 검사할 문자열의 맨 첫 번째 글자가 저장된 메모리 주소입니다. 반면 `std::strtod`는 문자를 읽다가 숫자가 아닌 글자를 만나는 순간 읽기를 포기하고 그 지점의 주소를 `endptr`에 심어줍니다.
* **단 1글자도 읽지 못한 상태 판정**: 만약 두 주소값이 완벽하게 일치한다면(`endptr == c_str()`), 이는 `strtod`가 첫 번째 문자부터 숫자가 아니라고 판단하여 **단 한 칸도 전진하지 못하고 즉시 파싱을 포기했음**을 의미합니다. 즉, 사용자가 숫자가 아닌 `"abc"`, `"hello"` 같은 100% 순수 문자열을 값 자리에 밀어 넣었을 때 이를 저수준 주소 비교 연산으로 가장 빠르고 정밀하게 포착해 내는 격리 장치입니다.

#### **Q7. 값이 너무 길어 `double` 그릇마저 터져버리는 초거대 숫자(예: 9가 100개 연속된 수)가 유입될 때, `Error: too large a number.` 가 정상 출력되는 원리는 무엇인가요?**

* **`HUGE_VAL` (무한대) 전환**: `std::strtod`는 숫자가 아무리 길어도 중간에 파싱을 포기하지 않고 널 문자(`\0`)까지 끝까지 읽어 나갑니다. 다만 `double` 타입의 최대 표현 범위($1.79 \times 10^{308}$)를 초과했음을 감지하면, 변수 값을 컴퓨터 수학상의 무한대를 뜻하는 상수인 **`HUGE_VAL` (`inf`)**로 채워버립니다.
* **무한대 대소 비교 연산**: 이후 코드는 범위 검사 식인 `if (value > 1000)` 문장을 만나게 됩니다. 컴퓨터 내부 연산에서 무한대(`inf`)는 당연히 `1000`보다 큰 수로 판정(`true`)되므로, 데이터가 오염되거나 `bad input`으로 오작동하는 일 없이 서브젝트가 요구하는 정확한 가이드라인인 **`Error: too large a number.`** 에러 궤도로 완벽하게 안착하여 처리됩니다.

#### **Q8. `std::atoi` 함수는 예외 처리가 불가능한 결함이 있는데, 본 로직의 연/월/일 파싱 부분에서 어떻게 안전하게 사용될 수 있었나요?**

* **상위 로직의 철저한 사전 검사**: `std::atoi`는 잘못된 문자열이 들어오면 오류를 감지하지 못하고 엉뚱하게 `0`을 뱉거나 오버플로우 시 쓰레기 값을 반환하는 약점이 있습니다.
* **완벽한 환경 조성**: 그러나 현재 코드는 `atoi`를 호출하기 전, 1단계 필터(`date.length() != 10`)를 통해 문자열 길이를 완벽히 제어하고, 2단계 필터(`std::isdigit` 루프)를 통해 날짜 포맷 내에 숫자가 아닌 모든 문자 예외 입력을 사전에 철저히 입구 컷 하고 있습니다. 결과적으로 `atoi` 내부로는 오직 검증된 범위의 순수 숫자 문자열 조각(`"2026"`, `"06"`)만 도달하므로, 함수 본연의 한계를 상위 계층의 다중 방어벽으로 완벽히 통제하여 절대적인 안정성을 보장합니다.

#### **Q9. 가장 가까운 과거 날짜를 찾기 위해 사용한 `lower_bound` 방식 대신 `std::map::upper_bound`만 사용하는 대안을 고려했을 때의 구조적 차이와 장단점은 무엇인가요?**

* **`upper_bound` 대안의 간결성**: `std::map::upper_bound(date)`는 입력 날짜보다 무조건 큰(`>`) 첫 번째 원소를 찾습니다. 일치하는 날짜가 있든 없든 상관없이, `upper_bound`가 반환한 반복자를 한 칸 뒤로 당기면(`--it`) 우리가 원하는 '당일 혹은 가장 가까운 과거 날짜'를 가리키게 됩니다. 이 방식을 채택하면 코드 맨 위에 위치한 `_data.find(date)` 일치 검사 구문을 통째로 생략할 수 있어 소스 코드가 비약적으로 간결해집니다.
* **`lower_bound` 명시적 조합의 의의**: 그럼에도 현재의 `find` + `lower_bound` 조합은 "정확히 일치하는 날짜가 존재할 때는 반복자 감소 연산(`--it`)을 단 한 번도 수행하지 않고 즉시 안전하게 값을 반환한다"는 **명시적인 직관성과 경계선 예외 안전성(Exception Safety)**을 제공합니다. 두 방식 모두 시간 복잡도는 $O(\log N)$으로 성능이 동일하므로, 개발자의 설계 스타일과 안정성 가치관에 따른 정당한 선택지입니다.

#### **Q10. `findRate` 멤버 함수 내부에서 `std::map`을 탐색할 때 일반 `iterator`가 아닌 `const_iterator`를 강제해야만 하는 컴파일 구조적 이유는 무엇인가요?**

* **`const` 함수 계약의 이행**: `findRate` 함수는 클래스의 내부 멤버 변수 데이터를 수정하지 않고 조회만 하겠다는 약속으로 함수 선언부 우측에 **`const` 키워드**가 마킹되어 있습니다.
* **읽기 전용 상태 전이**: 함수가 `const` 상태가 되면 내부 멤버 변수인 맵 컨테이너(`_data`) 역시 자동으로 '읽기 전용(Read-Only)' 상태로 엄격히 통제됩니다. 만약 내부 원소의 수정을 허용하는 일반 `iterator`로 맵을 탐색하려고 시도하면 컴파일러는 객체의 상수성 계약이 깨질 위험을 감지하고 즉시 컴파일 에러를 발생시킵니다. 따라서 데이터를 절대 오염시키지 않는 안전한 전용 나침반 포인터인 **`const_iterator`**를 사용하는 것이 C++ 정적 타입 시스템 명세에 부합하는 정석적인 설계입니다.
