## 🏛️ Exercise 00: Easy find

STL 알고리즘과 함수 템플릿의 기초를 결합하는 첫 번째 단계입니다. 다양한 정수형 순차 컨테이너 내부를 탐색하여 특정 값이 처음 등장하는 위치를 찾아내는 범용 함수 템플릿 `easyfind`를 설계합니다.

### Objectives

* 함수 템플릿 문법을 활용하여 컨테이너의 세부 타입에 종속되지 않고 작동하는 범용 탐색 로직 구축.
* 공통 인터페이스인 반복자(Iterator)와 표준 알고리즘인 `std::find`를 결합하여 STL의 핵심 설계 철학을 이해.
* 컨테이너 내에서 원하는 값을 찾지 못했을 때, 프로그램의 비정상 종료를 막고 상황을 통제할 수 있는 C++ 정석 예외 처리(Exception Handling) 메커니즘 숙지.

### 💡 Deep Dive

#### **Q1. 함수 내부에서 `T::iterator` 앞에 굳이 `typename`을 붙여주어야 하는 컴파일 구조적 이유가 무엇인가요?**

* **컴파일러의 딜레마**: 템플릿 함수 정의 시점에는 인자 `T`에 어떤 클래스가 들어올지 알 수 없습니다. 컴파일러가 코드를 읽을 때 `T::iterator`라는 문장을 만나면, 이것이 클래스 내부의 명백한 **'데이터 타입'**인지, 아니면 내부 선언된 static **'멤버 변수(값)'**인지 구별하지 못하는 모호함이 생깁니다.
* **C++ 표준 규칙의 방어**: C++ 규격상 앞에 아무것도 없으면 컴파일러는 이를 무조건 '변수(값)'로 취급하려고 시도합니다. 따라서 프로그래머가 맨 앞에 **`typename` 키워드를 명시하여 "이것은 변수가 아니라 뒤에 변수를 선언하기 위한 명백한 타입(Type)이다"라고 컴파일러에게 고지**해 주어야 컴파일 타임 1단계 구문 분석에서 에러 없이 올바른 구문 트리를 형성할 수 있습니다.

#### **Q2. `std::vector<int>`와 `std::list<int>`는 내부 구조가 완전히 다른데, 어떻게 동일한 `std::find` 알고리즘 하나로 둘 다 탐색할 수 있나요?**

* **반복자(Iterator)의 추상화**: `std::vector`는 메모리가 연속적인 동적 배열 구조이며, `std::list`는 메모리가 파편화되어 포인터로 엮인 이중 연결 리스트 구조입니다.
* **통일된 인터페이스**: STL 컨테이너들은 내부 구조와 무관하게 원소들을 순회할 수 있도록 포인터를 개념화한 객체인 **반복자(Iterator)**를 공통 인터페이스로 제공합니다. `std::find` 알고리즘은 컨테이너의 실체나 내부 메모리 배치 방식을 전혀 몰라도, 인자로 넘어온 `.begin()`과 `.end()` 반복자만을 이용해 똑같이 `++` 연산과 역참조(`*`) 연산으로 전진하며 비교할 수 있기 때문에 완벽한 다형성이 성립합니다.

#### **Q3. 커스텀 예외 클래스의 `what()` 함수 뒤에 붙은 `const throw()` 문법은 C++98 표준에서 어떤 의미를 갖나요?**

* **예외 지정자(Exception Specification)**: `throw()`는 **"이 함수는 실행 중에 내부에서 그 어떤 예외도 밖으로 던지지 않는다"**라고 컴파일러에게 약속하는 안전장치입니다. 예외 메시지를 반환하는 `what()` 함수 본문 안에서 또 다른 예외가 발생하면 프로그램이 터지는 대참사가 일어나므로 이를 원천 차단합니다.
* **정석 오버라이딩**: C++98 표준의 최상위 부모 예외 클래스인 `std::exception::what()` 함수에 `const throw()`가 이미 명시되어 있습니다. 따라서 부모의 가상 함수를 정상적으로 재정의(Overriding)하기 위해 자식 클래스의 `what()` 함수 선언부에도 반드시 똑같이 붙여주어야 컴파일 에러가 나지 않습니다.

#### **Q4. 과제문 가이드라인에 "연관 컨테이너(Associative Containers)는 처리할 필요가 없다"고 명시된 이유가 무엇인가요?**

* **데이터 구조 및 인터페이스의 불일치**: `std::map`이나 `std::set` 같은 연관 컨테이너는 값이 순서대로 저장되지 않고, `Key-Value` 쌍(`std::pair`)의 트리 구조로 정렬되어 저장됩니다. 따라서 정수(`int`) 값을 일대일로 순차 비교하는 `easyfind` 함수에 원소 타입이 다른 `map`을 집어넣으면 구조가 맞지 않아 컴파일 에러가 터집니다.
* **STL의 효율성 철학**: `std::find` 알고리즘은 처음부터 끝까지 다 뒤지는 **순차 탐색($O(N)$)**을 수행합니다. 그러나 연관 컨테이너는 내부적으로 정렬된 이진 트리 구조이기 때문에 고유의 **`.find()` 멤버 함수**를 내장하고 있으며, 이를 쓰면 **이진 탐색($O(\log N)$)**으로 비교할 수 없을 만큼 빠르게 값을 찾습니다. 이미 이진 탐색이 가능한 강력한 컨테이너를 $O(N)$짜리 순차 탐색 함수에 강제로 밀어 넣는 것은 STL의 효율성 철학에 위배되므로 과제 범위에서 제외된 것입니다.

#### **Q5. 예외 메시지를 화면에 출력할 때 `std::cout` 대신 `std::cerr`를 사용하는 기술적인 이점은 무엇인가요?**

* **버퍼링 유무(Unbuffered)**: `std::cout`은 출력 비용을 아끼기 위해 메모리 내 임시 저장 공간인 버퍼에 데이터를 모았다가 출력하는 **Buffered** 방식입니다. 반면 `std::cerr`는 데이터를 받는 즉시 모니터 화면에 내뿜는 **Unbuffered** 방식입니다. 만약 치명적인 에러로 프로그램이 강제 종료되더라도 버퍼에 갇혀 에러 로그가 유실되는 현상을 막고 즉각적으로 에러 상황을 인지시킬 수 있습니다.
* **표준 스트림 리다이렉션의 분리**: 리눅스 환경에서 프로그램의 정상 출력 결과를 파일로 저장하기 위해 리다이렉션 연산자(`./easyfind > output.txt`)를 사용할 때, `std::cout`으로 찍힌 문자열은 파일로 들어가지만 `std::cerr`로 찍힌 에러 메시지는 여전히 터미널 화면에 실시간으로 표시되므로 결괏값과 에러 로그를 명확하게 이원화하여 관리할 수 있습니다.

## 🏛️ Exercise 01: Span

STL 컨테이너와 알고리즘의 최적화 능력을 검증하고 대규모 데이터를 제어하는 클래스를 설계하는 단계입니다. 최대 `N`개의 정수를 저장할 수 있는 `Span` 클래스를 구축하고, 저장된 숫자들 사이의 최단 거리(`shortestSpan`) 및 최장 거리(`longestSpan`)를 연산하는 알고리즘을 구현합니다.

### Objectives

* Orthodox Canonical Class Form(OCCF)을 준수하는 견고한 일반 클래스 설계 및 소스 분리 규칙 숙지.
* 무작위 데이터 속에서 최소/최대 구간을 도출할 때, 시간 복잡도를 $O(N^2)$에서 $O(N \log N)$ 또는 $O(N)$으로 혁신하는 적절한 STL 알고리즘 조합 역량 배양.
* 단일 삽입의 한계를 넘어, 복사할 데이터의 원본 컨테이너 종류에 구애받지 않고 일괄 수용하는 반복자 범위 기반 템플릿 함수(`addNumbers`) 설계.
* 부호 있는 정수(`int`) 간의 연산에서 발생할 수 있는 오버플로우 한계를 `unsigned int`와 `static_cast` 및 컴퓨터 비트 연산의 특성을 이용해 수학적으로 방어하는 능력 강화.

### 💡 Deep Dive

#### **Q1. `addNumbers` 함수 선언 시 `template <typename InputIterator>`를 도입하고, 구현부까지 `.hpp`에 포함해야만 하는 구조적 이유가 무엇인가요?**

* **타입의 범용성 확보**: 만약 인자 타입을 `std::vector<int>::iterator`로 고정해 버리면, 외부 데이터가 `std::list`나 `std::deque`에 담겨 있을 때 `Span`에 일괄 삽입할 수 없는 폐쇄적인 구조가 됩니다. 템플릿 매개변수 `InputIterator`를 사용하면, 원본 컨테이너가 무엇이든 정수 알맹이를 지니고 반복자 인터페이스를 충족하기만 하면 전부 수용할 수 있는 STL 표준 알고리즘 스타일의 다형성이 확보됩니다.
* **템플릿 인스턴스화와 링킹 에러 방지**: 함수 템플릿은 호출 시점에 컴파일러가 실제 인자 타입을 바탕으로 구체적인 기계어 코드를 찍어내는 '인스턴스화(Instantiation)' 과정을 거칩니다. 만약 구현부를 `Span.cpp`로 격리하면 `main.cpp` 컴파일 시 알맹이가 보이지 않아 기계어 코드를 생성하지 못하고, 결국 링킹 단계에서 `undefined reference` 에러가 터집니다. 따라서 컴파일러가 호출부에서 설계도를 즉시 읽어 코드를 빌드할 수 있도록 선언과 구현 모두 헤더 파일(`.hpp`)에 포함시켜야 합니다.

#### **Q2. `std::distance(first, last)` 함수는 인자로 `std::vector`의 반복자가 들어올 때와 `std::list`의 반복자가 들어올 때 내부 작동 방식과 성능이 어떻게 달라지나요?**

* **반복자 카테고리에 따른 최적화 기법**: `std::distance`는 내부적으로 전달받은 반복자의 카테고리 태그(Iterator Category)를 판별하여 최적의 경로로 동작합니다.
* **`std::vector` (임의 접근 반복자, Random Access Iterator)**: 벡터는 메모리가 연속적인 배열 구조이므로 두 포인터 간의 단일 뺄셈 연산(`last - first`)을 수행합니다. 따라서 데이터의 양에 관계없이 **$O(1)$ (상수 시간)** 만에 거리를 즉시 계산해 냅니다.
* **`std::list` (양방향 반복자, Bidirectional Iterator)**: 리스트는 메모리가 파편화된 연결 리스트 구조이므로 직통 뺄셈이 불가능합니다. 내부적으로 `while` 루프를 돌며 `++first` 연산으로 노드를 끝까지 한 칸씩 타고 이동하며 개수를 세야 하므로 **$O(N)$ (선형 시간)**의 성능을 보이며 데이터 크기만큼 연산 시간이 늘어납니다.

#### **Q3. `_vec.insert(_vec.end(), first, last);` 범위를 활용한 삽입이 직접 루프를 돌며 `push_back`을 만 번 호출하는 것보다 압도적으로 효율적인 성능을 내는 원리가 무엇인가요?**

* **벡터의 메모리 재할당(Reallocation) 비용**: `push_back`을 루프로 돌리면 상자(Capacity)가 가득 찰 때마다 벡터는 힙 영역에 더 큰 메모리를 새로 잡고, 기존 원소를 전부 복사한 뒤 오래된 공간을 해제하는 대공사를 반복 전개합니다. 수만 개 데이터 삽입 시 이 오버헤드가 누적되어 심각한 병목을 유발합니다.
* **범위 삽입의 사전 용량 최적화**: 반면 `insert` 범위 메서드는 삽입을 감행하기 전 `std::distance`를 통해 들어올 데이터의 총량을 먼저 계산합니다. 만약 공간이 부족하더라도 필요한 용량만큼 **딱 1회만 메모리를 재할당**하여 상자를 키운 뒤, 루프 없이 메모리 통째로 블록 복사를 수행하기 때문에 불필요한 이사(복사) 비용을 완벽하게 제거하여 극적인 속도 향상을 이루어냅니다.

#### **Q4. 복사 대입 연산자(`operator=`)와 달리, 복사 생성자 내부에서는 `if (this != &other)`와 같은 자기 대입 검사를 수행하지 않는 이유가 무엇인가요?**

* **객체의 생명 주기와 존재 유무의 차이**:
* **복사 대입 연산자**: 이미 과거에 생성되어 정상 작동 중인 두 객체 간의 대입 연산입니다. 프로그래머의 실수나 포인터 조작 오류 등으로 `a = a;` 같은 자기 대입이 일어날 가능성이 존재하며, 이때 기존 자원을 무턱대고 해제했다간 자가 파괴(Self-destruction)가 일어나므로 주소 검사를 통한 방어벽이 필수적입니다.
* **복사 생성자**: 기존 객체를 기반으로 **새로운 객체를 최초로 창조하는 순간**에 호출됩니다. 호출 시점에는 `this` 객체가 메모리에 온전히 존재하지도 않던 상태이므로, 이미 존재하던 원본 객체(`other`)와 주소가 일치할 확률이 구조적으로 0%입니다. 따라서 불필요한 조건문 검사를 생략하는 것이 타당합니다.

#### **Q5. `Span sp = Span(5);` 구문과 `Span largeSp(10000);` 구문은 문법적으로 어떤 차이가 있으며, `=` 기호가 있음에도 복사 대입 연산자가 호출되지 않는 이유는 무엇인가요?**

* **초기화 기호와 대입 연산 기호의 구분**: C++에서 복사 대입 연산자(`operator=`)는 **이미 독자적으로 생성을 마친 두 객체** 사이에서만 동작합니다. 객체가 선언되는 문장 맨 처음에 등장하는 `=` 기호는 대입 연산이 아니라 단지 **"복사 초기화(Copy Initialization)"**를 뜻하는 문법적 기호일 뿐입니다.
* **컴파일러의 복사 제거(Copy Elision)**: `Span sp = Span(5);`는 엄밀한 문법상 우측에 임시 객체(Temporary Object)를 먼저 생성한 후 좌측 객체의 복사 생성자를 호출해야 하지만, 현대 컴파일러(C++98 플래그 포함)는 자원 낭비를 막기 위해 임시 객체 생성을 생략하는 '복사 제거' 최적화를 단행합니다. 결과적으로 직접 초기화 방식인 `Span largeSp(10000);`과 완전히 동일하게 생성자가 다이렉트로 호출되도록 리팩토링되어 실행됩니다.

#### **Q6. `shortestSpan()`에서 최단 거리를 탐색할 때 오름차순 정렬(`std::sort`)을 수행하면 시간 복잡도가 혁신적으로 단축되는 원리가 무엇인가요?**

* **$O(N^2)$에서 $O(N \log N)$으로의 대전환**: 무작위로 뒤섞인 배열 상태에서 가장 가까운 두 수의 거리를 찾으려면 이중 `for`문으로 모든 숫자 쌍을 비교해야 하므로 $O(N^2)$이 걸립니다. 데이터가 만 개일 경우 1억 번 연산해야 하므로 대규모 대량 데이터 환경에서 타임아웃을 유발합니다.
* **인접 원소 속성의 활용**: 데이터를 부하가 적은 하이브리드 알고리즘인 **`std::sort` ($O(N \log N)$)**로 정렬하고 나면, 수학적으로 **값이 가장 비슷한 두 수는 무조건 배열 상에서 바로 옆자리(인접)**에 밀착하게 됩니다. 정렬 후 단 1회만 단선 순회($O(N)$)하며 인접 원소 간의 차이만 검사하면 되므로 대규모 테스트에서도 눈 깜짝할 사이에 최단 거리를 정밀하게 검출할 수 있습니다.

#### **Q7. `shortestSpan()` 탐색 중 인접 원소 간의 차이가 `0`이 나오는 순간 루프를 탈출(`return 0;`)하도록 조기 종료(Early Exit) 조건을 추가해도 수학적/구조적으로 안전한가요?**

* **최소 간격 한계의 정당성**: `Span` 클래스에서 구하는 수치 간의 거리는 음수가 존재할 수 없는 절대 거리의 개념입니다. 거리가 가질 수 있는 완벽한 최솟값은 `0`이므로, 정렬된 배열을 훑다가 차이가 `0`(동일한 숫자 중복 입수)인 지점을 발견했다면 그 뒤를 아무리 뒤져도 `0`보다 작은 거리는 출현할 수 없습니다.
* **연산량 대폭 압축**: 대규모 난수 테스트(`std::rand()`) 진행 시 비둘기집의 원리에 의해 중복 숫자가 자주 유입됩니다. 이때 조기 종료 코드가 탑재되어 있다면 수천 번 남은 반복 연산 궤도를 단칼에 끊고 즉시 `0`을 반환하므로, 불필요한 자원 낭비를 완벽히 차단하고 실행 성능을 극대화할 수 있습니다.

#### **Q8. `longestSpan()`에서 `std::sort`를 배제하고 `std::min_element`와 `std::max_element`를 독립적으로 채택한 성능적 이점은 무엇인가요?**

* **알고리즘 목적에 맞는 복잡도 최적화**: 가장 긴 거리(Longest Span)는 전체 데이터 중 **`최댓값 - 최솟값`** 하나만 계산하면 즉시 유도됩니다. 이를 구하겠다고 배열 전체를 $O(N \log N)$ 비용을 들여 정렬하는 것은 심각한 자원 낭비입니다.
* **선형 탐색 최적화**: 범위 내를 정직하게 한 바퀴만 훑으며 가장 큰 값과 가장 작은 값의 위치(반복자)를 사냥해 오는 **$O(N)$ 성능의 `std::min_element`와 `std::max_element`** 알고리즘을 사용하면, 불필요한 정렬 오버헤드를 원천 배제하여 최장 거리를 가장 빠르고 날카롭게 뽑아낼 수 있습니다.

#### **Q9. 최단/최장 거리 연산 시 왜 굳이 `static_cast`를 거쳐 `unsigned int` 영역에서 뺄셈을 집행해야 하며, 극단적인 음수와 양수 대치 시 오버플로우가 제어되는 원리가 무엇인가요?**

* **부호 비트 손상 방지**: `int` 타입 상태로 최고값(`2,147,483,647`)에서 최저값(`-2,147,483,648`)을 빼버리면 표현 범위를 아득히 초과하여 부호 비트가 파괴되는 정수 오버플로우 쓰레기값이 도출됩니다.
* **2의 보수와 모듈러 연산의 마법**: C++ 규격상 `unsigned int` 연산은 절대 터지지 않고 시계처럼 회전하는 모듈러 산술(Modular Arithmetic)을 따릅니다. 최고값과 최저값을 컴퓨터 비트(2의 보수) 형태로 유지한 채 `static_cast<unsigned int>` 안경을 씌우면, `0x7FFFFFFF`와 `0x80000000`이라는 양수들로 재해석됩니다. 이 상태로 둘을 빼면(`0x7FFFFFFF - 0x80000000`) 언더플로우 비트 회전 메커니즘에 의해 정확히 `0xFFFFFFFF`가 도출되며, 이를 십진수로 환산하면 두 극단 값 사이의 정확한 수학적 절대 거리인 **`4,294,967,295`**가 유실 없이 완벽하게 복원됩니다.

#### **Q10. 최솟값 기준점을 잡을 때 `static_cast<unsigned int>(-1)` 이라는 기묘한 문법으로 변수를 초기화하는 이유와 기술적 혜택이 무엇인가요?**

* **우주 최대값(INF)의 정석적인 빌드**: `shortestSpan`에서 최솟값을 안전하게 누적 갱신(Min Update)해 나가려면, 초기 기준점 상자에는 그 어떤 비교 대상이 치고 들어와도 무조건 패배할 수밖에 없는 거대한 최대값(INF) 장벽이 먼저 충전되어 있어야 합니다.
* **비트 마스킹과 시스템 독립성**: 컴퓨터가 정수 `-1`을 기억하는 2의 보수 비트 형태는 모든 칸이 `1`로 꽉 찬 **`0xFFFFFFFF`**입니다. 이를 `static_cast<unsigned int>`로 컴파일러에게 밀어 넣으면, 부호 비트 빗장이 풀리며 `unsigned int` 자료형이 32비트 환경에서 도달할 수 있는 전지전능한 최대값인 **`4,294,967,295`**로 강제 바인딩됩니다. 직접 숫자를 타이핑하다 발생하는 오타 위험을 원천 차단하며, 향후 시스템 환경이 64비트 등으로 이식 및 변모하더라도 해당 시스템이 지원하는 한계치 최대값 비트를 컴파일러가 알아서 꽉 채워 초기화해 주기 때문에 코드의 안전성과 이식성이 비약적으로 상승합니다.

## 🏛️ Exercise 02: MutantStack

STL 컨테이너 어댑터의 설계 한계를 상속과 템플릿 문법을 활용해 우아하게 깨뜨려보는 최종 보스 단계입니다. 기존 `std::stack`이 제공하는 후입선출(LIFO)의 강력한 캡슐화와 편의성은 고스란히 계승하면서도, 원래는 접근이 불가능했던 내부 원소 순회용 반복자(Iterator) 인터페이스를 새로 탑재한 돌연변이 스택 `MutantStack`을 설계합니다.

### Objectives

* Orthodox Canonical Class Form(OCCF)을 준수하는 견고한 클래스 템플릿 설계 기법 완벽 체득.
* 기존 컨테이너를 포장하여 기능을 제한하는 **컨테이너 어댑터(Container Adapter)**의 설계 목적과 철학 이해.
* 부모 클래스가 지닌 `protected` 멤버 자원에 영리하게 접근하여 기능을 확장하는 객체지향 상속성 마스터.
* 템플릿 상속 관계에서 발생하는 컴파일러의 식별자 검색 한계를 `this->` 지시어와 `typename`, `typedef` 조합으로 해결하는 고급 문법 규격 숙지.

### 💡 Deep Dive

#### **Q1. `typedef typename std::stack<T>::container_type::iterator iterator;` 구문에서 `container_type`이란 무엇이며, `typename`은 왜 생략하면 안 되나요?**

* **`container_type`을 통한 범용성 확보**: `std::stack`은 스스로 데이터를 저장하지 않고 내부의 다른 순차 컨테이너에게 작업을 위임합니다. 이때 `std::stack` 내부 소스코드를 보면, 사용자가 지정한 내부 저장소 타입을 표준 명칭으로 추적할 수 있도록 `typedef Container container_type;`이라고 타입을 정의해 두었습니다. 즉, 별도 설정을 안 했다면 기본값인 `std::deque<T>`를 가리킵니다.
* **의존 타입(Dependent Type) 식별을 위한 `typename`**: 컴파일러가 이 줄을 파싱할 때 부모 클래스 `std::stack<T>`는 아직 `T`가 결정되지 않은 미정 상태입니다. 컴파일러 표준 규칙상 미정 템플릿에 의존하는 하위 이름은 앞에 아무것도 없으면 데이터 타입이 아니라 static '멤버 변수(값)'로 간주하려 시도합니다. 따라서 맨 앞에 **`typename` 키워드를 선언하여 "뒤에 오는 식별자는 변수가 아니라 명백한 '데이터 타입'이다"**라고 컴파일러에게 확답을 주어야 구문 분석 에러를 막을 수 있습니다.

#### **Q2. `begin()` 함수 구현부에서 부모의 컨테이너를 호출할 때 그냥 `c.begin()`이 아니라 굳이 `this->c.begin()` 형태로 `this->`를 강제 명시해야 하는 컴파일러적 이유는 무엇인가요?**

* **C++ 템플릿의 2단계 탐색(Two-Phase Lookup) 메커니즘**: 컴파일러가 템플릿 클래스를 읽을 때는 컴파일을 2단계로 나누어 진행합니다. 1단계(정적 분석 시점)에는 `T`가 무엇인지 모르는 상태에서 문법 규칙만 검사하고, 2단계(인스턴스화 시점)에 실제 타입을 만나면 기계어 코드를 새로 찍어냅니다.
* **이름 탐색의 지연**: 만약 자식 함수 안에서 `this->` 없이 그냥 `c`를 쓰면, 컴파일러는 이 식별자가 부모와 무관한 독립 변수라고 판단하고 자식 스코프와 전역 공간만 검색하다가 "찾을 수 없다"며 1단계에서 빌드를 터뜨립니다. 반면 **`this->c`라고 명시하면 컴파일러는 이 변수가 나 자신의 타입에 의존하는 이름임을 인지하여, 검색을 잠시 보류하고 2단계(인스턴스화 시점)로 미룹니다.** 이후 실제 타입이 결정되면 구체화된 부모 클래스 안으로 들어가 `protected` 멤버 변수인 `c`를 안전하게 찾아내어 연결해 줍니다.

#### **Q3. 복사 대입 연산자(`operator=`) 내부에서 `std::stack<T>::operator=(other);` 처럼 상위 클래스의 연산자를 명시적으로 강제 호출해 주어야 하는 구조적 이유가 무엇인가요?**

* **이름 가려짐(Name Hiding) 극복**: 자식 클래스인 `MutantStack`에서 대입 연산자를 새로 오버로딩하여 정의하는 순간, 상속 구조상 부모 클래스가 기본적으로 가지고 있던 `operator=` 함수들은 자식의 연산자 이름에 의해 가려져서 내부에서 일반적인 방식으로는 호출할 수 없게 차단됩니다.
* **자원 복사 책임의 올바른 위임**: 대입 연산자는 생성자와 달리 자식 클래스에서 재정의할 때 부모의 대입 연산자가 자동으로 연쇄 호출되지 않습니다. `MutantStack`은 독자적인 멤버 변수가 없고 모든 데이터 자원이 부모의 내부 컨테이너 `c`에 몰려 있으므로, 부모 호출을 생략하면 `a = b;`를 실행해도 데이터가 전혀 복사되지 않는 치명적인 버그가 생깁니다. 따라서 범위 지정 연산자(`::`)를 통해 **`std::stack<T>::operator=(other);`를 명시적으로 호출함으로써 부모 몫의 자원(내부 컨테이너 `c`)을 완벽하게 깊은 복사(Deep Copy)해 오도록 책임을 위임**하는 것이 객체지향 설계상 완벽한 정석입니다.

#### **Q4. 일반적인 클래스 상속과 달리, 부모 클래스인 `std::stack`에 가상 소멸자(`virtual`)가 부재함에도 자식 클래스에서 별도의 메모리 누수나 크래시 위험이 없는 비결이 무엇인가요?**

* **독자적 동적 할당 자원의 부재**: 업캐스팅된 부모 포인터 타입(`std::stack*`)으로 객체를 `delete`할 경우 가상 소멸자가 없다면 자식 소멸자가 호출되지 않는 미정의 동작(Undefined Behavior)이 발생합니다. 하지만 `MutantStack`은 자식 단계에서 새로 추가한 독자적인 포인터나 동적 할당 변수가 전혀 없기 때문에 자식 소멸자가 실행되지 않더라도 소멸자 누수 레이더에 걸릴 물리적 자원이 없습니다.
* **부모 소멸자의 연쇄 청소**: 자식 객체가 메모리에서 파괴될 때 자식 소멸자 호출 여부와 상관없이 부모 클래스인 `std::stack`의 소멸자는 최종 단계에서 무조건 정상 호출됩니다. 부모 소멸자가 실행되면 내부에 들고 있던 멤버 변수인 `c`(`std::deque`)의 소멸자 역시 연쇄적으로 호출되면서 데이터가 쌓여있던 힙 영역을 알아서 완벽하게 청소해 주기 때문에 메모리 구조상 안전합니다. (단, 다형성 포인터 형태로 동적 할당(`new/delete`)하여 쓰는 클래스가 아니라는 설계적 규칙을 지켜야 합니다.)

#### **Q5. `MutantStack` 클래스 템플릿 내부에서 4종 반복자(일반, 상수, 역방향, 상수 역방향)의 타입 정의(선언)와 `begin/end` 계열 멤버 함수(구현)의 구조적 톱니바퀴 관계는 어떻게 되나요?**

* **선언과 구현의 철저한 분리와 결합**:
  * **`typedef` 블록(선언)**: 설계도 최상단에 부모 내부 컨테이너가 숨겨둔 극도로 길고 복잡한 4가지 반복자 주소 명칭들을 가져와, 내 클래스 내부와 외부 사용자가 단 세 글자 혹은 직관적인 명칭으로 편리하게 사용할 수 있도록 사전 작업(별칭 선언)을 전개해 둔 곳입니다.
  * **`begin/end` 블록(구현)**: 선언부에서 개척해 놓은 축약 타입들(`iterator`, `const_iterator` 등)을 반환형으로 삼아, 진짜로 명령을 수행하고 비트를 넘겨주는 기계어 코드를 작성(구현)하는 곳입니다. 선언이 토대를 완벽하게 닦아주었기 때문에 `iterator begin() { return (this->c.begin()); }`과 같이 불필요한 중복 구문 없이 극도로 높은 가득성과 정돈된 구조를 유지할 수 있게 됩니다.

#### **Q6. 과제 PDF 기본 테스트 코드 중 `std::stack<int> s(mstack);` 구문을 실행했을 때, 변수 `s`를 대상으로 `s.begin()`과 같은 반복자 순회를 시도하면 컴파일 에러가 터지는 언어적 원리가 무엇인가요?**

* **복사 슬라이싱(Copy Slicing)의 한계**: `std::stack<int> s(mstack);` 코드는 자식 객체(`MutantStack`)를 부모 클래스 자료형 상자(`std::stack`)에 집어넣어 강제 복사하는 **업캐스팅** 과정입니다. 이 순간 복사 슬라이싱 현상이 발생하여 부모가 가진 알맹이 데이터 변수 `c`만 정상 복사되고, 자식이 가졌던 돌연변이 특수 기능(`begin/end` 등의 멤버 함수 및 인터페이스 인터프리터)은 칼로 자르듯 전부 깎여나가 소멸합니다.
* **정적 타입 시스템의 통제**: 생성된 변수 `s` 공식 타입은 순수한 부모 `std::stack`입니다. 원래 `std::stack` 표준 명세에는 `begin()`과 `end()`라는 식별자 자체가 아예 존재하지 않으므로, 컴파일러는 이를 감지하고 빌드를 거부합니다. 복사본의 원소들을 여전히 반복자로 제어하고 싶다면 받아주는 상자의 타입 역시 자식 형태인 `MutantStack<int> s(mstack);`으로 엄격히 일치시켜 주어야 합니다.

#### **Q7. 확장 반복자 테스트 구역의 `for (; rit != rite; ++rit)` 문법에서 첫 번째 초기식 자리를 과감하게 비워두어도 정상 작동하는 컴파일러적 메커니즘과 그에 따른 차이점은 무엇인가요?**

* **초기화 구문 생략의 합법성**: C++ `for` 문의 첫 번째 매개변수 영역(초기식)은 반드시 변수를 새로 선언해야 하는 강제 공간이 아닙니다. 이미 `for` 문이 시작되기 바로 전 윗줄 영역에서 `rit`라는 반복자 변수를 안전하게 선언하고 초기값(`mstack.rbegin()`)까지 정밀 충전해 두었기 때문에, 세미콜론(`;`)만 찍어 자리를 구분해 주면 컴파일러는 초기화 과정을 스킵하고 곧바로 2단계 조건식 검사 단계로 진입합니다.
* **변수 생명 주기(Scope)의 차이**: 일반적인 `for (int i = 0; ...)` 형태는 변수의 수명이 `for` 문의 중괄호(`{ }`) 안으로 격리되어 루프가 종료되는 순간 공중분해 됩니다. 반면 이와 같이 변수를 바깥으로 빼고 첫 칸을 비우는 기법을 적용하면, **`for` 문 루프 정산이 끝난 시점 아래에서도 해당 반복자(`rit`)가 최종적으로 머물렀던 위치나 값을 파괴 없이 그대로 유지하며 코드를 재사용**할 수 있다는 결정적인 수명 주기 관리 차이점이 존재합니다. 또한 C++98 환경에서 극도로 길어지는 타입명 때문에 발생하는 가로 가독성 저하 문제를 해결하고 소스 코드의 통일성을 지켜주는 순기능을 발휘합니다.
