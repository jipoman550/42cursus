# 📖 C++ - Module 03
**Inheritance (상속)**

## 📝 Introduction
이 프로젝트는 객체지향 프로그래밍(OOP)의 핵심 개념인 **상속(Inheritance)**을 학습합니다. 단순히 코드를 재사용하는 것을 넘어, 부모 클래스의 유산을 자식이 어떻게 물려받고 확장하는지, 그리고 다중 상속 시 발생하는 복잡한 설계 문제인 **다이아몬드 문제(Diamond Problem)**를 **가상 상속(Virtual Inheritance)**으로 어떻게 우아하게 해결하는지 깊이 있게 다룹니다.

## 🛠️ General Rules
과제를 수행하며 반드시 준수해야 하는 규칙들입니다.

### 1. Compilation
* **Compiler**: `c++`
* **Flags**: `-Wall -Wextra -Werror`
* **Standard**: `-std=c++98` 플래그를 추가해도 완벽하게 컴파일되어야 합니다.

### 2. Naming Conventions
* **Class Names**: **UpperCamelCase** 형식을 따릅니다.
* **Files**: 헤더 파일과 소스 파일명은 반드시 클래스 이름과 일치해야 합니다.

### 3. Forbidden & Allowed
* **C 함수 사용 금지**: `printf()`, `malloc()`, `free()` 사용 시 **0점** 처리됩니다.
* **STL 제한**: Module 08 이전까지 컨테이너와 알고리즘 헤더 사용이 금지됩니다.
* **Orthodox Canonical Class Form (OCCF)**: 모든 클래스는 반드시 기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자를 포함해야 합니다.
* **🚨 [NEW] Virtual Destructor**: 상속 관계에서 메모리 누수를 방지하기 위해 부모 클래스의 소멸자에 `virtual` 키워드를 반드시 사용해야 합니다.

## 🏗️ Exercise 00: Aaaaand... OPEN!
기본 클래스인 `ClapTrap`을 설계하며 캡슐화와 기본 로직을 구현하는 단계입니다.

### Objectives
* 이름, HP(10), EP(10), AD(0) 속성을 가진 `ClapTrap` 클래스 구현.
* 에너지와 체력이 0이 아닐 때만 동작하는 `attack`, `takeDamage`, `beRepaired` 함수 구현.

### 💡 Deep Dive

#### **Q1. 속성을 `private`에 두는 캡슐화(Encapsulation)의 이유는?**
* **답변**: 클래스 외부에서 `_hitPoints` 등에 직접 접근하여 말도 안 되는 값(예: -500)을 넣는 것을 막기 위함입니다. 속성은 내부(`private`)에 숨기고, 오직 허용된 함수(`public`)를 통해서만 값이 변하도록 통제하여 데이터의 무결성을 보호합니다.

#### **Q2. `unsigned int` 파라미터에 음수가 들어오면 어떻게 되나요?**
* **답변**: C++ 표준에 따라 암시적 형변환이 일어나 매우 큰 양수(예: 42억)로 해석됩니다. `takeDamage`에서는 이를 큰 데미지로 간주해 HP를 0으로 만들도록 로직을 설계해야 하며, 이는 자료형의 특성을 이해하고 방어적인 코드를 짜는 연습이 됩니다.

#### **Q3. `hp <= 0` 대신 `hp == 0` 체크를 써도 안전한 이유는?**
* **답변**: `unsigned int`는 태생적으로 0 미만의 값을 가질 수 없습니다. 또한 `takeDamage` 구현 시 `amount >= _hitPoints`인 경우를 미리 체크하여 `_hitPoints`를 강제로 `0`에 고정하는 **언더플로우 방지 로직**을 넣었기 때문에, 값이 음수가 되어 한 바퀴 도는 현상 없이 항상 정확히 0에 안착하게 됩니다.

## 🛡️ Exercise 01: Serena, my love!
`ClapTrap`을 상속받은 `ScavTrap`을 만들며 상속의 기초와 `protected` 접근 제어자를 학습합니다.

### Objectives
* `ClapTrap`을 상속받아 능력치가 강화된 `ScavTrap` 구현 (HP: 100, EP: 50, AD: 20).
* 생성자와 소멸자의 호출 순서(Chaining) 확인.
* 특수 능력 `guardGate()` (문지기 모드) 구현.

### 💡 Deep Dive

#### **Q1. `private`과 `protected` 상속의 결정적 차이는?**
* **답변**: `protected`는 "가족(자손들)끼리만 공유하는 유산"입니다. 부모의 멤버를 `protected`로 설정하면 자식 클래스 내부에서는 직접 접근이 가능하지만, 외부(`main` 등)에서의 접근은 여전히 차단됩니다. 반면 `private` 멤버는 자식조차 접근할 수 없습니다.

#### **Q2. 생성자와 소멸자의 호출 순서가 다른 이유는?**
* **답변**: **생성은 부모 → 자식**, **소멸은 자식 → 부모** 순서입니다. 자식이 살아있는 동안 부모의 기능(`_name` 등)을 계속 사용해야 하므로 부모가 먼저 태어나 기초를 닦고, 소멸할 때는 자식이 먼저 자신의 일을 정리한 뒤 뿌리가 되는 부모가 마지막에 사라지는 논리입니다.

#### **Q3. 부모 포인터로 자식을 가리킬 때(`Base* ptr = new Derived()`) 왜 가상 소멸자가 필요한가요?**
* **답변**: 이를 **업캐스팅**이라고 합니다. 소멸자가 `virtual`이 아니면 컴파일러는 포인터 타입인 부모 소멸자만 호출하고 끝내버려 자식 객체의 메모리가 해제되지 않습니다. `virtual`을 붙여야 실행 시점에 실제 객체(자식)를 확인하여 자식 소멸자부터 부모 소멸자까지 연쇄적으로 호출하게 됩니다.

## 💎 Exercise 03: Now it's weird!
다중 상속의 꽃인 `DiamondTrap`을 구현하며 다이아몬드 문제와 가상 상속을 정복합니다.

### Objectives
* `ScavTrap`과 `FragTrap`을 동시에 상속받는 `DiamondTrap` 구현.
* **다이아몬드 문제**를 해결하기 위한 **가상 상속(`virtual inheritance`)** 적용.
* 이름 쉐도잉(Name Shadowing) 및 `-Wshadow` 플래그 대응.

### 💡 Deep Dive

#### **Q1. 다이아몬드 문제와 '가상 상속(`virtual`)의 마법'은?**
* **답변**: 자식이 두 부모를 통해 조상(`ClapTrap`)을 중복 상속받아 메모리에 조상이 두 개 생기는 문제입니다. 부모들이 조상을 상속할 때 `virtual` 키워드를 쓰면, 최하위 자식인 `DiamondTrap` 안에 조상 인스턴스가 **단 하나만** 존재하도록 공유 메모리 구조를 형성합니다.

#### **Q2. 가상 상속 시 생성자 호출의 특이점은?**
* **답변**: 가상 상속 상태에서는 중간 부모(`ScavTrap`, `FragTrap`)가 아닌, 최하위 파생 클래스인 **`DiamondTrap`이 직접 조상(`ClapTrap`)의 생성자를 호출**할 권한과 책임을 가집니다. 그래서 네가 `ClapTrap(name + "_clap_name")`을 직접 호출하여 접미사 요구사항을 지킨 것입니다.

#### **Q3. `-Wshadow` 플래그와 이름 쉐도잉의 의미는?**
* **답변**: 자식 클래스에서 부모와 동일한 이름의 변수(`_name`)를 선언하면 부모의 변수가 가려지는 현상입니다. 과제는 이를 의도적으로 요구하며, 컴파일러가 경고를 내뱉을 때 `-Wno-shadow`로 이를 허용하거나 `ClapTrap::_name` 처럼 명시적으로 범위를 지정해 두 변수를 구분하는 능력을 평가합니다.
