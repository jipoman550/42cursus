# 📖 C++ - Module 07

**C++ Templates (C++ 템플릿)**

## 📝 Introduction

이 프로젝트는 코드의 재사용성을 극대화하고 타입에 종속되지 않는 범형 프로그래밍(Generic Programming)의 핵심인 **C++ 템플릿(Templates)**의 개념과 올바른 구현체를 다룹니다. 단순한 코드 복사-붙여넣기 방식의 매크로를 탈피하여, 컴파일 타임에 안전하게 타입을 바인딩하고 다형성을 확장하는 방법을 학습합니다.

## 🛠️ General Rules

과제를 수행하며 반드시 준수해야 하는 규칙들입니다.

### 1. Compilation

* **Compiler**: `c++`
* **Flags**: `-Wall -Wextra -Werror`
* **Standard**: `-std=c++98` 플래그를 추가해도 완벽하게 컴파일되어야 합니다.

### 2. Forbidden & Allowed

* **C 함수 사용 금지**: `printf()`, `malloc()`, `free()` 사용 시 **0점** 처리됩니다.
* **사용 금지 키워드**: `using namespace`와 `friend` 사용 시 **-42점** 감점됩니다.
* **Orthodox Canonical Class Form (OCCF)**: 템플릿 클래스 역시 반드시 기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자를 구현해야 하며, 템플릿 특성에 맞게 설계되어야 합니다.
* **구현부의 위치**: 템플릿은 컴파일러가 호출 시점에 코드를 인스턴스화해야 하므로, 일반적인 `.cpp` 분리가 아닌 **헤더 파일(`.hpp` 또는 `.hpp` 내부에 포함된 `.tpp`)에 선언과 구현이 함께 존재**해야 합니다.

## 🏛️ Exercise 00: Start with a few functions

가장 직관적이고 널리 쓰이는 함수 템플릿(Function Template)의 기초를 다집니다. 임의의 두 데이터 타입을 안전하게 교환하는 `swap`, 더 큰 값을 반환하는 `max`, 더 작은 값을 반환하는 `min`을 설계합니다.

### Objectives

* 함수 템플릿의 정의 문법(`template <typename T>`) 숙지 및 인수 추론(Argument Deduction) 메커니즘 이해.
* 단순 대입 연산자가 아닌, 객체의 깊은 복사(Deep Copy)를 고려한 안전한 데이터 교환 logic 구축.
* 두 인자의 값이 완전히 동일할 때, 과제 PDF 규칙에 따라 두 번째 인자를 정확히 반환하는 엄격한 비교 연산 조건 처리.

### 💡 Deep Dive

#### **Q1. `max` 나 `min` 함수를 구현할 때 `a > b ? a : b` 대신 굳이 인자의 참조자(`const T&`)를 받고 참조자를 반환하는 이유가 무엇인가요?**

* **메모리와 비용의 최적화**: 인자를 일반 변수(`T a`)로 받으면 함수가 호출될 때마다 무거운 객체(예: 거대한 구조체나 복잡한 클래스)의 복사 생성자가 실행되어 스택 메모리와 CPU 자원을 낭비합니다.
* **참조를 통한 원본 유지**: `const T&`를 사용하면 읽기 전용으로 원본 객체의 주소만 슥 건네받으므로 복사 비용이 "0원"이 됩니다. 반환 시에도 참조로 넘겨주어야 연속적인 연산이나 대입에서 불필요한 임시 객체 생성을 막을 수 있습니다.

#### **Q2. `max(a, b)`에서 두 값이 같을 때 왜 굳이 '두 번째 인자'를 반환하라고 서브젝트에 명시되어 있나요? 어차피 값은 똑같지 않나요?**

* **값의 동일성과 객체의 동일성**: 눈에 보이는 수치나 연산 결과(`operator==`)가 같더라도, 메모리 상에 존재하는 두 객체 `a`와 `b`는 엄연히 다른 인스턴스입니다.
* **엄격한 규칙 평가**: 42과제의 채점 기준과 PDF 예시 규격은 매우 깐깐합니다. `a >= b` 조건으로 검사하면 같을 때 `a`를 반환하게 되므로, 출제자의 의도에 완벽히 부합하려면 `a > b` 조건문이나 `b >= a` 형태를 취해 **수치가 같을 때 무조건 우측에 입력된 인자(`b`)가 반환**되도록 디펜스 코드를 정밀하게 짜야 합니다.

## 🏛️ Exercise 01: Iter

임의의 타입으로 구성된 배열의 모든 요소에 사용자가 정의한 임의의 함수를 순차적으로 적용하는 범용 순회 함수 템플릿 `iter`를 설계합니다.

### Objectives

* 배열의 시작 주소, 배열의 길이, 그리고 각 요소에 대응할 함수 포인터(또는 함수 템플릿 인스턴스)를 유연하게 바인딩하는 능력 배양.
* 함수 포인터 역시 템플릿 인자(`typename F` 또는 명시적 함수 포인터 타입)로 추론하여 타입 안정성을 극대화하는 설계 기법 습득.

### 💡 Deep Dive

#### **Q1. `iter` 함수의 3번째 인자인 함수 포인터를 선언할 때 `void (*f)(T&)`와 `void (*f)(const T&)` 중 어떤 것을 선택해야 하나요?**

* **함수의 목적에 따른 분리**: `iter`에 넘겨질 도우미 함수가 단순 출력용(예: `print`)이라면 원본 데이터를 훼손하지 않는 `const T&` 포맷이 안전합니다. 반면, 배열 내부의 값을 직접 수정하는 함수(예: `increment`)라면 `T&`가 필요합니다.
* **오버로딩을 통한 방어벽**: 가장 완벽한 C++ 템플릿 설계는 두 버전 모두에 유연하게 대응할 수 있도록 함수 오버로딩을 고려하거나, 함수 자체를 완전히 또 다른 템플릿 인자 `typename F`로 추론하게 만들어 컴파일러가 알아서 알맞은 함수 형식을 매핑하도록 위임하는 것입니다.

#### **Q2. 템플릿으로 만든 `print` 함수를 `iter`의 인자로 넘겨주려고 하니 컴파일 에러가 발생합니다. 일반 함수는 잘 되는데 왜 이러죠?**

* **실체가 없는 유령 상태**: 템플릿 함수(예: `template <typename T> void print(T& x)`) 그 자체는 컴파일러가 읽는 '설계도'일 뿐, 메모리에 물리적으로 존재하는 진짜 함수가 아닙니다.
* **명시적 인스턴스화의 필요성**: 컴파일러에게 어떤 타입의 `print`를 쓸지 힌트를 주지 않고 주소만 넘기면, 컴파일러는 `iter` 내부에서 추론을 포기하고 에러를 냅니다. 따라서 인자로 넘길 때는 `iter(array, length, print<int>);` 처럼 **함수 이름 뒤에 타겟 타입을 명시적으로 박아넣어서 설계도를 진짜 기계어 코드로 실체화(Instantiation)** 시켜준 뒤 전달해야 합니다.

## 🏛️ Exercise 02: Array

기본 자료형 배열이 가진 런타임 경계 검사 부재(Buffer Overflow) 문제를 완벽히 해결하고, 메모리를 동적 할당하여 관리하는 유틸리티 템플릿 클래스 `Array`를 구현합니다.

### Objectives

* 클래스 템플릿(Class Template) 내부에 동적 메모리 배열(`new T[]`)을 안전하게 캡슐화하는 기법 마스터.
* 깊은 복사(Deep Copy)를 보장하는 OCCF(기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자)의 정석적인 메모리 해제 및 재할당 로직 구축.
* 첨자 연산자(`operator[]`) 오버로딩 시, 범위를 벗어난 인덱스 접근을 감지하여 `std::out_of_range` 예외를 던지는 방어적 프로그래밍 구현.
* 수정 가능한 일반 객체용 `operator[]`와 읽기만 가능한 상수 객체용 `const operator[] const`를 분리 설계하는 디테일 습득.

### 💡 Deep Dive

#### **Q1. `Array.hpp` 파일과 `Array.cpp` 파일로 나누어 코딩한 뒤 컴파일했더니 `undefined reference` 링킹 에러가 발생합니다. 소스 코드는 완벽한데 왜 링커가 못 찾죠?**

* **컴파일러의 빌드 메커니즘**: 일반적인 `.cpp` 파일은 각각 독립적으로 컴파일되어 오브젝트 파일(`.o`)로 변환됩니다. 그러나 템플릿은 `main.cpp`에서 `Array<int> a;` 처럼 **실제 타입과 결합하여 호출하는 그 순간에 컴파일러가 해당 타입용 클래스 코드를 바이너리로 찍어냅니다.**
* **링커의 비명**: 만약 구현부를 `Array.cpp`에 꽁꽁 숨겨두면, `main.cpp`를 컴파일할 때 컴파일러는 설계도의 뼈대만 보고 껍데기만 만들어 둔 채 넘어갑니다. 나중에 링킹 단계에서 진짜 알맹이 코드를 찾으려고 하면 `Array.cpp` 쪽에서는 `int`용으로 변환된 코드가 생성된 적이 없기 때문에 "코드가 없다!"며 빌드가 터지는 것입니다.
* **해결책**: 템플릿 클래스는 과감하게 **선언과 구현 모두를 `.hpp` 파일에 때려 박아야 합니다.** 만약 가독성을 위해 분리하고 싶다면 구현부를 `.tpp` 파일에 작성한 뒤, `.hpp` 맨 하단에서 `#include "Array.tpp"` 형태로 강제 병합시켜 컴파일러가 한 번에 읽을 수 있도록 동선을 짜야 합니다.

#### **Q2. 복사 대입 연산자(`operator=`)를 구현할 때 기존에 들고 있던 메모리를 그냥 놔두고 `elements = fixed_array.elements` 형태로 대입하면 어떻게 되나요?**

* **치명적인 메모리 누수와 댕글링 포인터**: 주소값만 슥 복사하는 '얕은 복사(Shallow Copy)'의 전형적인 대참사 시나리오입니다.
* **메모리 고립**: 기존에 내가 동적 할당(`new`)해서 쥐고 있던 메모리 상자의 방 번호를 잃어버리게 되어, 힙(Heap) 메모리에 쓰레기가 평생 잔존하는 메모리 누수가 발생합니다.
* **이중 해제(Double Free)**: 더 끔찍한 것은, 두 객체가 동일한 메모리 주소를 가리키게 되면서 한 객체가 소멸자에서 `delete[] elements;`를 실행해 버리면, 나머지 남은 객체의 포인터는 이미 파괴된 방을 가리키는 유령 포인터(`Dangling Pointer`)가 됩니다. 이 상태에서 남은 객체도 죽을 때 똑같은 주소를 한 번 더 `delete` 하려고 하므로 즉시 `Double Free Corrupt` 크래시를 내며 프로그램이 폭발합니다. 반드시 기존 메모리를 `delete[]`로 안전하게 수거한 뒤, 새 크기만큼 `new`로 상자를 짜서 알맹이를 하나하나 깊은 복사해야 합니다.

#### **Q3. 첨자 연산자(`operator[]`)를 만들 때 굳이 함수를 두 버전(`T& operator[]` 와 `const T& operator[] const`)으로 나누어 똑같은 내용을 두 번 적어야 하는 이유가 무엇인가요?**

* **상수 객체의 권리 보장**: 만약 함수 내부에서 멤버 변수를 절대 바꾸지 않는 상수 객체 `const Array<int> c_arr(5);`가 존재한다고 가정해 봅시다. 이 객체에서 `c_arr[2]`를 호출하면 컴파일러는 객체의 상수성을 훼손하지 않을 안전한 함수를 찾습니다.
* **컴파일 타임의 거부**: 만약 `const` 버전의 연산자가 준비되어 있지 않다면, 컴파일러는 "이 연산자는 내부 값을 바꿀지도 모르는 위험한 함수다"라며 상수 객체의 접근을 원천 차단하고 컴파일 에러를 뿜어냅니다. 반대로 일반 객체는 값을 수정할 수 있어야 하므로 `T&`를 반환해 `arr[1] = 42;`가 가능하도록 길을 열어주어야 합니다. 즉, **읽기/쓰기 모드(`T&`)와 읽기 전용 모드(`const T& const`)의 선로를 컴파일러에게 각각 깔아주는 것**이 정석입니다.

#### **Q4. 빈 배열을 생성할 때(`Array<int> empty;`) 내부 포인터를 `NULL`로 세팅하는 것과 `new T[0]`으로 할당하는 것 중 어떤 것이 42과제 룰에 안전한가요?**

* **안전한 기본값의 선택**: C++ 표준 규격상 `new T[0]`은 문법적으로 완벽히 허용되며 고유한 포인터 주소를 반환합니다. 하지만 할당된 크기가 없기 때문에 이 주소를 역참조하는 것은 불가능합니다.
* **방어적 설계**: 과제에서 빈 배열 생성 시 예외 처리나 메모리 안정성을 직관적으로 통제하려면 내부 포인터를 `NULL`로 초기화하고 크기(`_size`)를 `0`으로 세팅하는 것이 훨씬 깔끔합니다. 이후 `operator[]`가 실행될 때 인덱스가 0 이상이거나 `_size`보다 크면 `std::out_of_range` 예외를 던지도록 사전에 차단막을 치면, `NULL` 포인터를 건드려 터지는 런타임 세그폴(Segfault) 에러를 완벽하게 원천 봉쇄할 수 있습니다.

## 📁 Directory Structure

* `ex00/`: 범용 데이터 교환 및 비교를 위한 함수 템플릿 (`swap`, `min`, `max`).
* `ex01/`: 배열 요소를 순회하며 다형성 기능을 적용하는 함수 템플릿 (`iter`).
* `ex02/`: 동적 할당 메모리와 엄격한 경계 검사 예외 처리가 결합된 유틸리티 클래스 템플릿 (`Array`).
