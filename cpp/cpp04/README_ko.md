# 📖 C++ - Module 04
**Subtype Polymorphism, Abstract Classes, and Interfaces (다형성, 추상 클래스, 인터페이스)**

## 📝 Introduction
이 프로젝트는 객체지향 프로그래밍(OOP)의 꽃이라 불리는 **다형성(Polymorphism)**을 깊이 있게 다룹니다. 부모 타입의 포인터로 자식 객체를 제어하는 **동적 바인딩(Dynamic Binding)**의 원리를 이해하고, 힙 메모리를 소유한 객체의 **깊은 복사(Deep Copy)**, 그리고 인스턴스화를 제한하는 **추상 클래스(Abstract Class)**와 **인터페이스(Interface)** 설계법을 학습합니다.

## 🛠️ General Rules
과제를 수행하며 반드시 준수해야 하는 규칙들입니다.

### 1. Compilation
* **Compiler**: `c++`
* **Flags**: `-Wall -Wextra -Werror`
* **Standard**: `-std=c++98` 플래그를 추가해도 완벽하게 컴파일되어야 합니다.

### 2. Forbidden & Allowed
* **C 함수 사용 금지**: `printf()`, `malloc()`, `free()` 사용 시 **0점** 처리됩니다.
* **STL 제한**: Module 08 이전까지 컨테이너(`vector`, `list`, `map` 등)와 알고리즘 사용이 금지됩니다.
* **Orthodox Canonical Class Form (OCCF)**: 모든 클래스는 반드시 기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자를 포함해야 합니다.
* **🚨 Virtual Destructor**: 다형성을 사용하는 상속 구조에서는 부모 클래스의 소멸자를 반드시 `virtual`로 선언하여 자식 객체의 메모리 누수를 방지해야 합니다.

## 🐾 Exercise 00: Polymorphism
부모 클래스 `Animal`과 자식 클래스 `Dog`, `Cat`을 통해 기본적인 다형성을 구현합니다.

### Objectives
* `virtual` 키워드를 사용한 `makeSound()` 함수의 동적 바인딩 확인.
* `WrongAnimal` 예시를 통해 `virtual`이 없을 때 발생하는 정적 바인딩 문제 학습.

### 💡 Deep Dive

#### **Q1. `Animal* meta = new Dog();` 에서 `virtual`이 왜 생사(生死)를 결정하나요?**
* **동적 바인딩(Dynamic Binding)**: `virtual`이 있으면 컴파일 시점이 아닌 **실행 시점(Runtime)**에 실제 객체(`Dog`/`Cat`)의 vtable을 참조하여 함수를 호출합니다.
* **가상 소멸자**: 부모 소멸자가 `virtual`이 아니면 `delete` 시 자식 소멸자가 호출되지 않아 **정의되지 않은 동작(UB)**이 발생하고, 자식이 별도의 메모리를 가질 경우(예: `Brain`) 누수로 이어집니다.

#### **Q2. 함수 뒤의 `const`와 `main`의 `const` 포인터는 어떤 관계인가요?**
* **Const 계약(Contract)**: `const Animal*` 포인터는 "객체를 수정하지 않겠다"는 약속입니다. 따라서 이 포인터는 "객체를 수정하지 않음"을 보증하는 **`const` 멤버 함수**만 호출할 수 있습니다.
* **오버라이딩 실패 주의**: 부모는 `const`인데 자식이 `const`를 빼고 구현하면, 시그니처가 달라 오버라이딩이 되지 않고 부모의 함수가 호출되는 결과(다형성 상실)를 초래합니다.

#### **Q3. 가상 함수 테이블(vtable)은 어떻게 동작하나요?**
* **설계도와 주소록**: 컴파일 타임에 클래스별로 가상 함수 주소록(vtable)이 생성됩니다. 자식 클래스가 오버라이딩을 하면 해당 칸의 주소가 자식 함수의 주소로 교체됩니다.
* **단일 점프**: 생성자처럼 부모부터 순차적으로 호출되는 것이 아니라, 실행 시점에 객체의 `vptr`을 타고 테이블로 가서 자식의 함수로 **단번에 점프**하여 실행합니다.

#### **Q4. 왜 굳이 부모 타입 포인터로 자식 객체를 다루나요? (Upcasting)**
* **관리의 효율성**: `Animal*` 배열 하나로 수많은 종류의 자식 객체들을 한꺼번에 관리(`makeSound()`, `delete` 등)할 수 있습니다.
* **확장성**: 새로운 동물 클래스가 추가되어도 기존의 관리 로직(함수, 배열 등)을 수정할 필요가 없는 유연한 설계를 가능하게 합니다.

## 🧠 Exercise 01: I don't want to set the world on fire
객체가 다른 객체를 포인터로 소유할 때 발생하는 메모리 관리와 복사 문제를 다룹니다.

### Objectives
* 100개의 아이디어를 가진 `Brain` 클래스 구현.
* `Dog`와 `Cat`이 `Brain`을 동적 할당하여 소유하도록 설계.
* **깊은 복사(Deep Copy)**를 통한 독립적인 객체 상태 유지.

### 💡 Deep Dive

#### **Q1. 얕은 복사(Shallow Copy)와 깊은 복사(Deep Copy)의 결정적 차이는 무엇인가요?**
* **얕은 복사**: `brain` 포인터의 **주소값**만 복사합니다. 이 경우 두 객체가 하나의 뇌를 공유하게 되어, 한 객체가 생각을 바꾸면 다른 객체도 영향을 받으며, 소멸 시 이미 해제된 메모리를 또 해제하려는 **이중 해제(Double Free)** 에러를 일으킵니다.
* **깊은 복사**: `new Brain(*other.brain)`을 통해 **별도의 메모리 공간**을 할당하고 내용물만 복사합니다. 두 객체는 완전히 독립된 상태를 유지하며 안전하게 소멸할 수 있습니다.

#### **Q2. 복사 대입 연산자에서 `delete this->brain;`을 수행하는 이유는 무엇인가요?**
* **메모리 누수 방지**: 대입 연산자는 이미 생성된 객체에 새로운 값을 덮어쓰는 과정입니다. 대상 객체(`this`)가 이미 가지고 있던 뇌를 먼저 메모리에서 해제(`delete`)하지 않고 새로운 뇌를 할당받으면, 기존 메모리의 주소를 잃어버려 **메모리 누수(Memory Leak)**가 발생합니다.
* **낡은 공책 비유**: 새로운 일기장(`other.brain`)을 복사해 오기 전에, 내 책상 위에 있던 낡은 공책(`this->brain`)을 먼저 버리는 과정과 같습니다.

#### **Q3. 자기 자신 대입 체크(`if (this != &other)`)가 왜 필수적인가요?**
* **객체 자살 방지**: 만약 `a = a;`와 같은 자기 자신 대입이 일어날 때 이 체크가 없다면, 자신의 뇌를 먼저 `delete` 해버린 후, 이미 삭제된 그 뇌를 참조하여 새로운 뇌를 만들려 시도하게 됩니다. 이는 **이미 해제된 메모리 접근(Segmentation Fault)**으로 이어집니다.

#### **Q4. 소멸자 호출 시 `Brain`이 `Dog`/`Cat`보다 먼저 찍히는 이유는 무엇인가요?**
* **명시적 호출 순서**: `Dog`의 소멸자 내부 **첫 번째 줄**에 `delete this->brain;`을 작성했기 때문입니다. `Dog` 소멸자가 시작되자마자 멤버 객체인 `Brain`의 소멸자가 먼저 호출되어 뒷정리를 끝내고, 그 후 `Dog` 자신의 메시지가 출력됩니다.
* **철거 원칙**: 안전한 자원 해제를 위해 항상 **자식(나중에 생성된 것) → 부모(먼저 생성된 것)** 순서로 철거가 진행됩니다.

#### **Q5. `new Brain(*other.brain)`에서 역참조(`*`)를 사용하는 이유는 무엇인가요?**
* **시그니처 일치**: `other.brain`은 포인터(`Brain*`) 타입입니다. `Brain` 클래스의 복사 생성자는 포인터가 아닌 **객체(`const Brain&`)**를 인자로 받도록 설계되어 있습니다.
* 따라서 포인터를 역참조하여 실제 '뇌 객체'를 꺼내 넘겨주어야만 내용물을 복사하는 깊은 복사가 이루어집니다.

## 🚫 Exercise 02: Abstract class
"그냥 동물"은 존재할 수 없다는 논리적 설계를 위해 추상 클래스를 도입합니다.

### Objectives
* `Animal` 클래스를 인스턴스화할 수 없는 **추상 클래스**로 변환.
* **순수 가상 함수(Pure Virtual Function)** 문법 이해.

### 💡 Deep Dive

#### **Q1. 왜 추상 클래스(Abstract Class)가 필요한가요?**
* **논리적 결함 해결**: 우리 주변에 '개'나 '고양이'는 존재하지만, 구체적인 종이 없는 "그냥 동물"은 존재하지 않습니다.
* **설계적 장치**: `Animal` 클래스를 직접 객체화(Instantiate)하지 못하게 막음으로써, 부모 클래스가 단지 자식들을 묶어주는 '개념'으로만 존재하도록 강제하는 것입니다.

#### **Q2. 추상 클래스로 만드는 가장 확실한 방법은 무엇인가요?**
* **순수 가상 함수(Pure Virtual Function)**: 가상 함수 선언 뒤에 `= 0;`을 붙여 구현부 없이 선언만 남깁니다.
* 하나 이상의 순수 가상 함수를 포함하는 클래스는 자동으로 추상 클래스가 되며, 직접적인 객체 생성이 원천 봉쇄됩니다.

#### **Q3. 만약 자식 클래스(`Dog`, `Cat`)에서 순수 가상 함수를 구현하지 않으면 어떻게 되나요?**
* **추상성의 상속**: 자식 클래스가 부모의 순수 가상 함수를 오버라이딩하여 구현하지 않으면, 자식 클래스 또한 '미완성된 설계도'를 가진 것으로 간주되어 **추상 클래스**가 됩니다.
* 이 경우 해당 자식 클래스 역시 객체를 생성할 수 없으며, 컴파일러가 에러를 통해 구현을 강제하게 됩니다.

#### **Q4. 추상 클래스가 된 `Animal`의 생성자는 여전히 호출되나요?**
* **암시적 호출(Implicit Call)**: 네, 자식 객체가 생성될 때 부모인 `Animal`의 생성자는 명시하지 않더라도 **컴파일러에 의해 자동으로 호출**됩니다.
* **순서 보장**: 객체 지향의 원칙에 따라 "기초(부모)가 먼저 준비되어야 자식(객체)이 존재할 수 있기 때문"에 항상 부모 생성자가 먼저 실행됩니다.

## 💎 Exercise 03: Interface & recap
C++98에서 인터페이스 개념을 구현하고, 복잡한 객체 생성 시스템을 설계합니다.

### Objectives
* `AMateria`를 기반으로 한 `Ice`, `Cure` 클래스 구현.
* `ICharacter` 인터페이스를 통한 캐릭터 인벤토리 시스템 관리.
* `IMateriaSource`를 통한 마테리아 템플릿 생산(Prototype 패턴 기초).

### 💡 Deep Dive

#### **Q1. `use(int, ICharacter&)` 함수가 타겟을 전달해야 한다는 것은 무슨 의미인가요?**
* **위임과 다형성**: 캐릭터의 `use` 함수는 인벤토리 슬롯을 확인한 뒤, 그 슬롯에 있는 마테리아의 `use` 함수를 호출하며 매개변수로 받은 타겟을 그대로 전달해야 합니다.
* 이를 통해 캐릭터는 자신이 사용하는 마테리아가 무엇인지 몰라도, 마테리아 객체가 스스로 타겟에게 적절한 효과(Ice는 공격, Cure는 치유)를 적용하게 만드는 **동적 바인딩**을 실현합니다.

#### **Q2. 인터페이스(I)와 추상 클래스(A)를 굳이 나누어 설계하는 이유는 무엇인가요?**
* **인터페이스(I)**: "무엇을 할 수 있는가"에 대한 **강력한 약속(계약)**입니다. 캐릭터와 상점이 어떤 기능을 가졌는지 정의하여 객체 간의 결합도를 낮춥니다.
* **추상 클래스(A)**: "어떤 공통 유전자를 가졌는가"에 대한 **기초 설계**입니다. 마테리아들의 공통 속성(type)과 동작(clone)을 한곳에 모아 중복을 제거하고 논리적 일관성을 유지합니다.

#### **Q3. `unequip()` 시 왜 `floor` 배열 같은 별도의 관리가 필요한가요?**
* **메모리 누수 방지**: 과제 규칙상 `unequip`은 마테리아를 `delete` 해서는 안 됩니다. 하지만 주소값을 잃어버리면 메모리 누수가 발생하므로, 버려진 아이템들의 주소를 `floor` 배열에 보관했다가 캐릭터가 소멸할 때 한꺼번에 정리해야 합니다.

#### **Q4. 마테리아를 장착할 때 왜 `tmp` 포인터를 거쳐서 `equip()`을 호출하나요?**
* **안전한 메모리 해제**: `createMateria`는 항상 `new`로 새로운 객체를 반환합니다. 만약 인벤토리가 가득 차서 `equip`에 실패할 경우, `tmp`에 주소를 받아두지 않으면 해당 객체를 `delete` 할 방법이 없어 누수가 발생하기 때문입니다.

#### **Q5. "Any type of AMateria"를 지원한다는 말은 어떤 문법적 원리를 뜻하나요?**
* **업캐스팅(Upcasting)**: 캐릭터의 인벤토리가 부모 타입인 `AMateria*` 배열로 선언되어 있음을 뜻합니다. 부모 포인터는 자식 객체(`Ice`, `Cure` 등)를 모두 가리킬 수 있으므로, 새로운 마테리아가 추가되어도 캐릭터 코드를 수정할 필요가 없는 확장성을 제공합니다.

#### **Q6. 인터페이스의 순수 가상 소멸자 옆에 왜 `{}`가 붙어 있나요?**
* **링킹 에러 방지**: C++ 규칙상 소멸자는 아무리 할 일이 없어도(순수 가상 함수일지라도) 반드시 몸체(정의)가 존재해야 합니다. 자식 객체가 소멸할 때 부모 소멸자를 호출하기 때문에, 별도의 `.cpp` 파일이 없는 인터페이스에서는 헤더에 빈 몸체 `{}`를 인라인으로 제공하는 것입니다.

#### **Q7. `learnMateria`에서 슬롯이 가득 찼을 때 왜 `delete`를 해줘야 하나요?**
* **소유권 책임**: `new`로 생성된 객체의 주소를 인자로 받은 함수는, 그 객체를 보관함에 저장하지 못했을 경우 책임을 지고 메모리를 해제해야 합니다. 그렇지 않으면 해당 주소를 알고 있는 유일한 통로가 사라져 "Definitely lost" 누수가 발생합니다.
