# Picoshell: Minimal Pipeline Implementation

**Picoshell**은 C 언어를 사용하여 UNIX의 파이프라인(`|`) 메커니즘을 최소 단위로 구현한 학습용 프로젝트입니다. `pipe`, `fork`, `dup2`, `execvp` 등 시스템 콜의 유기적인 흐름을 이해하는 데 목적이 있습니다.

---

## 🚀 주요 기능

* **다중 파이프라인 지원**: `ls | grep | wc`와 같이 무제한에 가까운 명령어 연결이 가능합니다.
* **프로세스 릴레이 시스템**: 이전 자식 프로세스의 출력을 다음 자식의 입력으로 정밀하게 전달합니다.
* **자원 관리 및 동기화**: 좀비 프로세스 방지를 위한 `wait` 처리 및 FD(File Descriptor) 누수 방지 로직이 포함되어 있습니다.

---

## 🏗 핵심 메커니즘: 릴레이 구조 (The Relay)

이 프로젝트의 핵심은 **"과거의 유산(`prev_read_end`)"**과 **"미래를 위한 준비(`pipe_fds`)"**를 연결하는 것입니다.

### 1. 자식 프로세스의 역할 (Relay Runner)

각 자식 프로세스는 루프 안에서 다음과 같은 작업을 수행합니다.

* **입력 방향 전환**: `prev_read_end`가 존재한다면(`-1`이 아니라면), 자신의 `STDIN`을 이전 파이프의 출구로 연결합니다.
* **출력 방향 전환**: 다음 명령어(`cmds[i+1]`)가 존재한다면, 새 파이프를 생성하고 자신의 `STDOUT`을 새 파이프의 입구로 연결합니다.
* **실행**: `execvp`를 통해 새로운 프로그램을 로드합니다.

### 2. 부모 프로세스의 역할 (Coordinator)

부모는 자식을 생성한 후 바톤 터치를 준비합니다.

* 사용이 끝난 `prev_read_end`를 닫아 FD 누수를 막습니다.
* 방금 생성한 파이프의 출구(`pipe_fds[0]`)를 `prev_read_end`에 저장하여 다음 루프의 자식에게 넘겨줍니다.

---

## 💡 주요 개념 정리

### 왜 `wait()`가 필수인가?

`ft_popen`이나 `picoshell`처럼 자식 프로세스를 생성하는 경우, 자식이 종료되어도 커널은 그 종료 상태를 보관합니다. 부모가 `wait`를 통해 이 정보를 수거하지 않으면 시스템 리소스를 낭비하는 **좀비 프로세스**가 발생합니다. 본 프로젝트는 모든 명령 실행 후 루프를 돌며 자식들을 회수합니다.

---

## 🛠 실행 예시

```c
char *cmd1[] = {"ls", "-l", NULL};
char *cmd2[] = {"grep", "picoshell", NULL};
char *cmd3[] = {"wc", "-l", NULL};

char **pipeline[] = {cmd1, cmd2, cmd3, NULL};

picoshell(pipeline);
// 실행 결과: ls -l | grep picoshell | wc -l 과 동일한 결과 출력

```

## 📝 배운 점 (Takeaways)

* **FD 관리의 엄격함**: 파이프의 쓰기 끝(`pipe_fds[1]`)을 정확히 닫지 않으면 읽기 쪽 프로세스가 EOF를 받지 못해 무한 대기에 빠질 수 있음을 학습했습니다.
* **프로세스 생명 주기**: `fork` 이후 부모와 자식의 메모리 공간은 분리되지만, 열려 있는 FD 테이블은 공유된다는 점을 이용해 IPC를 구현했습니다.
